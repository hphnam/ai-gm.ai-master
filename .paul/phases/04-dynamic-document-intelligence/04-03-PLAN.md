## Goal
Procedural docs become first-class. Every DocumentType carries a `kind` (`reference` | `procedural`). When a KnowledgeItem lands with a procedural DocumentType (via matched path or owner-accepted proposal), a `ChecklistExtractorService` pass extracts structured `{ steps, schedule, audience }` into a new `Checklist` row. Completion tables ship as empty-but-ready schema so 04-04 scheduler + 04-05 walkthrough have persistence primitives to build on.

## Purpose
CONTEXT.md vision: "AI as literal general manager." 04-02 gave every tenant a per-org taxonomy. This plan is the pivot from "taxonomy of doc types" → "procedural docs the assistant can *execute*, not just retrieve." The Checklist entity is the routing gate 04-04 (scheduler+WhatsApp) and 04-05 (walkthrough runtime) both consume. Without this plan's schema + extraction, the next two plans have nothing to target.

## Design decisions locked at plan-time (resolving CONTEXT.md "Open for planning")
- **Checklist ↔ KnowledgeItem cardinality:** 1-1 via `Checklist.knowledgeItemId @unique`. Simplest mental model. One KI → at-most-one procedural interpretation. Re-extraction on same KI replaces the existing row (handled via upsert). D-04-03-I tracks versioning if operators start asking "why did this week's steps change?"
- **Completion-state tables ship now, empty.** `ChecklistInstance` + `ChecklistStepCompletion` have no runtime writers this plan — they exist so 04-04 scheduler has a dedup surface (`(checklistId, instanceKey)` unique) and 04-05 walkthrough has a step-by-step persistence target. Lands now because (a) splitting the migration across plans invites drift and (b) both models are pure schema with zero risk to the existing /docs flow.
- **Extraction trigger:** fires when `documentTypeId` lands non-null AND the FK'd `DocumentType.kind === 'procedural'`. Applies to BOTH the classifier's matched path (on upload) AND the accept-type path (on owner confirmation of a procedural proposal). Proposal-pending rows do NOT extract — kind is unconfirmed, would waste Claude cost.
- **Schedule structure:** `ScheduleSchema` has a structured `cadence` enum (`daily | weekly | monthly | shift-start | shift-end | ad-hoc | unknown`) + `timeOfDay` (HH:MM nullable) + `dayOfWeek` (0-6 nullable) + `dayOfMonth` (1-31 nullable) + `rawText` (always present — "every Monday morning"). `.passthrough()` preserves emergent keys. Timezone is implicit single-tz-POC (UK) — D-04-03-D for tz support.
- **Step shape:** `ChecklistStepSchema` has `index` (ordinal), `text`, `kind` (`tick | numeric | photo | text`), `required` bool, `hint` nullable. `.passthrough()` preserves emergent keys. Photo step persists ref only (S3 migration = D-04-03-B).
- **Audience shape:** `AudienceSchema` has `roles` (array of `staff | manager | owner`, max 3) + `rawText` ("opening staff") + notes. User-level targeting = D-04-03-E.
- **Fail-soft extraction:** Claude 5xx / timeout / parse failure → no Checklist row, KnowledgeItem still ingests. `docs.checklist_extract_call` log fires with `result: 'error' | 'empty'` to surface the miss. Retry-extract UI = D-04-03-F.
- **No reclassify-flip existing rows:** only NEW uploads extract. If an operator flips an existing reference-type doc to procedural via D-04-02-I (rename/merge — not this plan), no back-extraction. D-04-03-C tracks retroactive extract if tenants ask.
- **Classifier cost envelope:** extractor is a second Claude call per procedural upload (~$0.01-0.05 incremental). Kept synchronous — same reasoning as 04-02 (acceptable added latency at POC scale). Async path = D-04-03-H (re-uses D-04-02-M scope trigger).
- **Mixed-format docs:** no special handling this plan. The classifier still sees the full extracted content and picks a kind. If the kind is `procedural` but content is prose-heavy, extractor may return an empty steps array — treated as fail-soft, no Checklist row. D-04-03-G tracks if this becomes noisy.

## <!-- audit-added --> Enterprise-audit runtime safety decisions (2026-04-21)
- **Extraction timeout is 30s** (matches classifier's 15s × 2 for bigger max_tokens budget). AbortController wires into `messages.create({ signal })`. Timeout counts against queue-timeout in semaphore (total upper bound per upload: 15s queue + 30s call = 45s ceiling).
- **Concurrency cap at 3 in-flight extractors** (mirror `MAX_CONCURRENT_IMAGE_EXTRACTS` at `apps/api/src/modules/docs/extractors/image-extractor.ts:35`). Prevents a burst of procedural uploads from wedging a pool of Anthropic calls. `docs.checklist_extract_queued` log fires on queue entry.
- **Step index normalization is load-bearing.** `ChecklistStepSchema` parses `index` as a number, but Claude may emit non-contiguous indices or duplicates. Post-parse `.map((s, i) => ({ ...s, index: i }))` normalizes to `[0..N-1]`. 04-05 walkthrough's current-step pointer assumes contiguity — breaking this guarantee here = breaking 04-05's state machine later.
- **ChecklistInstanceKey format is a contract between 04-03 (schema shipper) and 04-04 (scheduler writer).** The unique index `@@unique([checklistId, instanceKey])` doesn't constrain FORMAT, only uniqueness. Without the Zod regex contract shipped in Task 1, 04-04 could invent arbitrary formats and `ChecklistService.getInstance(checklistId, instanceKey)` lookups would silently miss.
- **Audit trail requires actingUserId on extraction.** `docs.checklist_extract_call` logs the mechanics; `docs.checklist_extracted` logs the accountability outcome. Both ship. audit-M5 boundary: extracted step text / schedule rawText / audience rawText stay OUT of both logs — only counts + enums + ids.
- **Kind-override needs an audit flag** (S5). `docs.type_accepted` previously emitted `name` only. With owner able to flip classifier's proposed kind, the log must carry `kind` + `kindOverridden` so an auditor can reconstruct "did owner override the classifier or accept as-proposed?"
- **Migration apply uses diff+deploy pattern** inherited from 04-01/02 (S1). `pnpm --filter @gm-ai/database exec prisma migrate dev` is interactive-TTY-only under Prisma 7 and will hang in non-interactive APPLY; the established `prisma migrate diff --from-config-datasource --to-schema --script` + hand-authored SQL + `prisma migrate deploy` pattern is the only supported path.
