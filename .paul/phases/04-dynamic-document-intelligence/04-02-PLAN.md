<objective>
## Goal
Build the document-shaping layer: every `/docs` upload passes through a Claude-based classifier that (a) matches the tenant's existing `DocumentType` rows with a confidence score, or (b) proposes a new type with a suggested schema. The owner confirms each proposal inline — no cluster-and-promote automation yet, no dedicated taxonomy-inbox page. Plus: fix the 04-01 leftovers (D-04-01-K) as prep.

## Purpose
Phase 4 CONTEXT.md: "Ingest any doc, figure out what KIND it is — reference / procedure / form / schedule / policy / anything else. Taxonomy evolves from the docs themselves, not from our release cycle." This plan is the first concrete step — it gives every tenant a per-org taxonomy built from their own uploads, owner-gated. No hardcoded doc types. Plans 04-03/04/05 consume `DocumentType.id` to gate procedural-vs-reference runtime behavior, so this taxonomy layer is load-bearing for the rest of Phase 4.

## Output
- Shared `DocumentType` Prisma model (organization-scoped) with `schema Json` for flexible per-tenant shapes.
- `KnowledgeItem.documentTypeId` (nullable link to confirmed type) + `KnowledgeItem.pendingTypeProposal Json?` (classifier's suggestion awaiting owner action).
- `ClassifierService.classify(content, orgId, title)` — Claude-based, returns `{ kind: 'matched', typeId, confidence } | { kind: 'proposal', name, description, schema, confidence }` or `{ kind: 'none' }` (for genuinely-unstructured content).
- `docs.classifier_call` structured log with tokens/usd/matchedExistingCount — no document content in logs.
- `POST /docs/:id/accept-type` + `POST /docs/:id/reject-type` endpoints.
- Inline owner-confirmation modal on successful upload when `pendingTypeProposal` is populated.
- `/docs` list shows type badge for classified rows + pending indicator with Accept/Reject buttons for unresolved proposals.
- 04-01 leftovers cleared: `@types/express` resolves the 7 TS7016 errors; `isZipHeader` signature accepts `Uint8Array` so the xlsx Buffer<ArrayBufferLike> TS2345 clears; drift migration adds the missing ChatConversation venueId FK + index and drops the stale `organizations.updatedAt` DB-side default.
</objective>

## Design decisions locked at plan-time (resolving CONTEXT.md "Open for planning")
- **Taxonomy storage:** single `DocumentType` Prisma model with `schema Json` — no per-tenant tables. Matches the `.passthrough()` agentic pattern (PROJECT.md Key Decisions 2026-04-18). Per-tenant tables deferred as D-04-02-A (trigger: first tenant whose schema stability + query performance justifies the migration overhead).
- **Promotion lifecycle:** owner hand-accepts every proposal. No embedding cluster-and-promote this plan. Deferred as D-04-02-B (trigger: taxonomy size on any org exceeds ~30 confirmed types AND duplicate-rename events become frequent). Keeps the loop synchronous, no BullMQ/Redis.
- **Confidence threshold for auto-classify:** `CLASSIFIER_AUTO_ACCEPT_CONFIDENCE = 0.7`. Above → `kind: 'matched'` auto-assigns `documentTypeId`; below → proposal path (owner decides). Constant lives in classifier service; not env-tunable this plan (D-04-02-C if a tenant needs per-org tuning).
- **No taxonomy-inbox settings page.** Pending proposals render inline on `/docs` with per-row Accept/Reject controls. Simpler UX for POC tenant volume; dedicated page is D-04-02-D (trigger: inbox surface exceeds ~10 simultaneous pending proposals on any org).
- **Scope stays in one plan (not split).** Task breakdown is 3 tasks; matches PAUL 2-3-task target.
- **XLSX multi-sheet handling:** unchanged from 04-01 — each uploaded XLSX → one `KnowledgeItem` with concatenated sheets. Per-sheet split remains D-04-01-E (trigger unchanged: 04-02 classifier reveals per-sheet treatment materially improves classification).
- **Classifier cost envelope:** no per-tenant monthly cap this plan. `docs.classifier_call` log emits tokens/usd/mime/duration. Hard cap = D-04-02-E (trigger: monthly aggregate classifier spend > $10/org).
