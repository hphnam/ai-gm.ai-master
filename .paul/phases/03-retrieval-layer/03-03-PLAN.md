<objective>
## Goal
Ship the retrieval half of the Agentic Knowledge Layer: a `KnowledgeRetrievalService` that does honest vector search over `knowledge_items` (returns `{ ok: false, reason: 'no-data' }` when nothing crosses the similarity threshold, instead of hallucinating a match), plus a `MockOpsService` that exposes the typed ops queries Phase 4's chat orchestration will call as Claude tools (`getStockBelowPar`, `getStockByName`, `getSupplierByName`, `getUpcomingCutoffs`). All results ride a single `ToolResult<T>` discriminated-union envelope published from `@gm-ai/types`, so every caller (chat engine, probe, future controllers) handles not-found the same way.

## Purpose
CONTEXT.md's "honest no-data" and "Claude picks which tools to call" goals are load-bearing for the demo — if retrieval returns low-quality vector hits dressed up as answers, or if an ops tool fakes a response on empty data, the whole "investor-ready" trust story collapses. Plan 03-02 proved agentic ingest produces rich metadata; 03-03 is what lets that metadata actually reach a conversation turn. Phase 4 chat orchestration (Plan 04-01) cannot start without both services existing and sharing the ToolResult envelope — no envelope means every chat tool wiring invents its own not-found shape, and we end up normalising later.

Closing this plan also closes Phase 3.

## Output
- `@gm-ai/types` — `ToolResult<T>` discriminated union + matching Zod schema (`ok: true, data: T` | `ok: false, reason, detail?`), exported from dist. Reused by retrieval + all mock-ops tools.
- `KnowledgeRetrievalService` (NestJS) in `apps/api/src/modules/retrieval/` — `find(query, opts)` that embeds via `EmbeddingsService.embedText` (inputType query), runs pgvector cosine, and returns `ToolResult<RetrievalHit[]>`; falls to `{ ok: false, reason: 'no-data' }` when top similarity < threshold.
- `MockOpsService` (NestJS) in `apps/api/src/modules/mock-ops/` — four methods, each returning `ToolResult<T>`: `getStockBelowPar(venueId)`, `getStockByName(venueId, name)`, `getSupplierByName(name)`, `getUpcomingCutoffs(venueId, withinHours?)`. Each honest about empty results.
- `probe-retrieval.ts` in `src/scripts/` — bootstraps a `RetrievalModule` + `MockOpsModule` Nest context, exercises: (a) a real-content query hits, (b) a nonsense query returns `no-data`, (c) each mock-ops tool returns a well-shaped envelope including a `no-data` path for at least one tool. Idempotent (no DB writes).
- `pnpm -w build` clean; `pnpm --filter api probe:retrieval` exits 0; `pnpm probe:seed` + `pnpm probe:ingest` still green (no regression).
</objective>

    ```ts
    import { z } from 'zod'

    export const TOOL_RESULT_REASONS = ['no-data', 'not-supported', 'error'] as const
    export type ToolResultReason = (typeof TOOL_RESULT_REASONS)[number]

    export type ToolResult<T> =
      | { ok: true; data: T }
      | { ok: false; reason: ToolResultReason; detail?: string }

    export function ok<T>(data: T): ToolResult<T> {
      return { ok: true, data }
    }

    export function fail(reason: ToolResultReason, detail?: string): ToolResult<never> {
      return { ok: false, reason, detail }
    }

    export function toolResultSchema<T extends z.ZodTypeAny>(dataSchema: T) {
      return z.discriminatedUnion('ok', [
        z.object({ ok: z.literal(true), data: dataSchema }),
        z.object({
          ok: z.literal(false),
          reason: z.enum(TOOL_RESULT_REASONS),
          detail: z.string().optional(),
        }),
      ])
    }
    ```
