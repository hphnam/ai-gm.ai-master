<objective>
## Goal

Close the **aggregate-query gap** on tabular CSV/XLSX documents. Today, Phase 1 retrieval surfaces a similarity-ranked slice of rows; the model cannot answer questions like "top 3 selling wines", "total revenue this month", or "highest priced item" because no compute layer ranges over the full table. This plan adds a structured-data path alongside the embedded-text path: at ingest, every row of a CSV/XLSX is tee'd into a JSONB row store; the agent gets a `query_document_table` tool with structured params (filter / group_by / aggregate / sort / limit); Postgres JSONB operators do the math.

## Purpose

Connects to PROJECT.md core requirement: *"Answer stock and ordering questions using live database data (stock levels, par levels, usage rates, purchase orders)"*. Stock/sales/ordering data arrives as spreadsheets — without an aggregation path, the assistant can do lookup but cannot do ranking, totals, or trends. This is the smallest fix that unlocks a class of questions GMs actually ask, without throwing graph/walkthrough work off-schedule (Phases 2-4 unaffected).

## Output

- New Prisma models `TabularRow` + `TabularColumn` with cascade-delete FK to `KnowledgeItem`
- New `tabular-extractor.ts` returning structured `{ columns: string[], rows: Record<string,string>[] }` for csv + xlsx mimes (parallel to existing string-returning extractors — no signature change)
- Ingest tee in `ingest.service.ts` persisting rows + inferred column types alongside existing section persistence
- New `TabularModule` + `TabularQueryService` with structured-DSL query builder over JSONB (no raw SQL surface)
- New agent tool `query_document_table` registered alongside `find_knowledge` in `ai-sdk-tools.ts` + `tool-dispatcher.ts`
- System-prompt nudge in `system-prompt.ts` directing the agent to use `query_document_table` on BOTH whole-table question shapes over tabular sources: aggregate/ranking ("top 3", "total") AND enumeration ("list all steps", "walk me through")
- New `probe-tabular.ts` script with ≥17 assertions (ingest fidelity, type inference, query correctness across all aggregate fns, enumeration shape with `_row_index` source-order sort + `truncated` flag, cross-tenant isolation, large-doc 5000-row smoke, hard-cap behaviour, idempotent re-ingest)
- New Zod schemas in `@gm-ai/types/tabular` for `TabularQueryInput` (filter / group_by / aggregate / sort / limit shape) + tool I/O contracts
</objective>

## AC-3: Naive column-type inference — number / date / string with majority-vote
```gherkin
Given a CSV with columns ["sku" (all strings), "price" (all numeric), "ordered_at" (all ISO-date), "notes" (mixed)]
When `inferColumnTypes(rows)` runs at ingest time
Then "sku" → 'string', "price" → 'number', "ordered_at" → 'date', "notes" → 'string'
And inference uses majority-vote: a column is 'number' if ≥80% of non-empty cells parse via `Number()` (excluding NaN), 'date' if ≥80% parse via `Date.parse()` and are not also valid numbers, otherwise 'string'
And empty / null cells are excluded from the vote (do not count toward majority either way)
And ambiguous columns (e.g. all empty, or below 80% threshold) default to 'string'
And the inferred_type is stored on `tabular_columns` for use by the query tool's auto-cast logic
```

## AC-4: query_document_table tool — structured DSL, deterministic aggregates
```gherkin
Given a tabular doc with 100 rows of {name, qty, price} where some `name` values repeat
When the agent calls `query_document_table({ doc_id, group_by: 'name', aggregate: { column: 'qty', fn: 'sum' }, sort: { column: '_aggregate', direction: 'desc' }, limit: 3 })`
Then the response contains the top-3 names by SUM(qty) with deterministic ordering (qty desc, name asc tie-break)
And SUM is computed via `(data->>'qty')::numeric` because `tabular_columns.inferred_type='number'`
And every aggregate fn { count, sum, avg, min, max } produces correct results on a fixture-verified test row set
And filter ops { eq, gt, lt, gte, lte, contains } produce correct row subsets
And tool input is validated by `TabularQueryInputSchema` (Zod) — invalid column names / unknown ops return ToolResult with `ok:false` reason='invalid-input', NOT a 500
And the tool NEVER constructs raw SQL from user-supplied strings — column names are validated against `tabular_columns.name` whitelist before composition; ops are mapped from a closed enum to fixed SQL fragments
```

    **Step 1 — `TabularQueryInputSchema` in `@gm-ai/types/tabular`**: Zod schema with shape:

    ```ts
    {
      docId: string (uuid)
      filters?: Array<{ column: string; op: 'eq'|'gt'|'lt'|'gte'|'lte'|'contains'; value: string|number }>
      groupBy?: string
      aggregate?: { column?: string; fn: 'count'|'sum'|'avg'|'min'|'max' }
      sort?: { column: string; direction: 'asc'|'desc' }  // column may be '_aggregate' (sort by aggregate result) or '_row_index' (sort by source-row position — used for enumeration / "list all rows in order")
      limit?: number (default 100, max 1000)
    }
    ```

    Closed enums for `op` and `fn`. Magic sort columns: `_aggregate` (only valid when an aggregate is present) and `_row_index` (always valid — maps to `tabular_rows.row_index`). Output type `TabularQueryResult = { rows: Array<Record<string, string|number>>; rowCount: number; truncated: boolean }` (truncated=true when the result was capped by `limit`).

    **Step 2 — `TabularModule`**: New NestJS module at `apps/api/src/modules/tabular/`. Exports `TabularQueryService`. PrismaModule import.

    **Step 3 — `TabularQueryService.query(orgId: string, input: TabularQueryInput): Promise<ToolResult<TabularQueryResult>>`**:
    - First fetch `KnowledgeItem.organizationId` for `input.docId`. If mismatch with `orgId` (or row missing), return `{ ok: false, reason: 'not-found' }` and emit `tabular.cross_org_denied` log with hashed orgId.
    - Fetch `tabular_columns` for `docId` → build whitelist of valid column names + their inferred types.
    - Validate every column reference in input (filter columns, groupBy, aggregate.column, sort.column except magic columns `_aggregate` and `_row_index`) against the whitelist. Reject with `{ ok: false, reason: 'invalid-input', details: { unknownColumn } }` if any column name not in whitelist. `_aggregate` is only valid when an aggregate is present (else reject); `_row_index` is always valid and resolves to `tabular_rows.row_index`.
    - Build a parameterized SQL query via `prisma.$queryRaw` using JSONB operators:
      - SELECT: if groupBy → `(data->>'$groupBy') AS group_key, $aggExpr AS aggregate`; else if aggregate (no groupBy) → just `$aggExpr AS aggregate`; else → `data`.
      - FROM `tabular_rows tr JOIN knowledge_items ki ON tr.doc_id = ki.id WHERE ki.organization_id = $1 AND tr.doc_id = $2` (non-bypassable tenant guard, separate from the whitelist check above — defence in depth).
      - WHERE: AND-chained filters with op mapping to fixed fragments (`(data->>'col')::numeric > $n` for number-typed `gt`/`lt`/`gte`/`lte`; `data->>'col' ILIKE '%' || $n || '%'` for `contains`; `data->>'col' = $n` for `eq` etc.). Cast `::numeric` only when `tabular_columns.inferred_type='number'`; otherwise string-equality.
      - GROUP BY: if groupBy.
      - ORDER BY: per `sort` — `_aggregate` resolves to the SELECT alias; `_row_index` resolves to `tr.row_index`; otherwise the column name (validated against whitelist, expressed as `(data->>'col')` or `(data->>'col')::numeric` per inferred type). If `sort` is omitted AND no groupBy AND no aggregate (the enumeration shape), default to `ORDER BY tr.row_index ASC` so "list all rows" is deterministic and matches the source document's row order.
      - LIMIT: per `limit` (default 100, hard-capped at 1000). When the result is truncated by the cap, surface `truncated: true` in the response so the agent can communicate "showing first N of M" rather than silently dropping rows.
    - All user-supplied values bound as parameters via `Prisma.sql` template tags. Column names interpolated only AFTER whitelist validation. Op + fn fragments come from a closed map `{ eq: '=', gt: '>', ... }` — no string concat with user input.
    - Aggregate fns:
      - `count` → `COUNT(*)`
      - `sum/avg/min/max` → `SUM((data->>'$col')::numeric)` etc. — REQUIRES `tabular_columns.inferred_type='number'` for `$col`; otherwise reject with `{ ok: false, reason: 'invalid-input', details: { reason: 'aggregate-requires-numeric-column' } }`.

    **Step 4 — Logging**: Every successful query emits `tabular.queried` log with `{ docId, orgIdHash, opCount, rowsReturned, latencyMs }` — NEVER row content. Cross-tenant denials emit `tabular.cross_org_denied`.

    Avoid: ANY raw SQL string concat with user input. Avoid: skipping the org-guard via "convenience" — even if the column whitelist already filters per docId, the JOIN + WHERE on `ki.organization_id` is non-negotiable defence-in-depth (Phase 1 SOC-2 CC6.6 carry-forward).
