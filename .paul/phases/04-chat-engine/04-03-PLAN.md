<objective>
## Goal
Ship the adaptation loop: persist `MessageFeedback` (thumbs-up/down/regenerate) against assistant messages, auto-enqueue low-similarity retrievals + thumbs-down/regeneration targets onto a `ReTagQueueItem` table, drain the queue by re-ingesting affected `KnowledgeItem` rows through `IngestService`, and run a canned eval harness that scores retrieval quality across a fixed query set.

## Purpose
Closes the POC's learning loop (roadmap Phase 4 final plan). Today the system has honest no-data (03-03) and proactive composition (04-02) but no signal back from real interactions — a wrong answer and a right answer look the same to the DB. This plan wires user feedback + implicit low-similarity signals into a queue that IngestService drains by re-authoring metadata, and adds a canned eval harness so we can measure whether re-tagging actually improves retrieval.

## Output
- Two new Prisma models: `MessageFeedback` (1:1 with assistant `ChatMessage`) + `ReTagQueueItem` (dedupe-on-active-status) applied via `prisma migrate diff → deploy` (Phase 3 approval-gated pattern).
- `AdaptationService` with `captureFeedback`, `enqueueReTag`, `processReTagQueue`, `captureRetrievalOutcome` — wired into `ChatService.sendMessage` post-persist for the low-similarity path.
- `packages/types/src/adaptation.ts` — `FeedbackKind`, `ReTagReason`, `ReTagStatus`, Zod schemas.
- `apps/api/src/scripts/probe-adaptation.ts` (E2E feedback → queue → process) + `apps/api/src/scripts/probe-eval.ts` (canned query set with per-query expected provenance, pass-threshold report).
- Named observability events: `adaptation.feedback_captured`, `adaptation.retag_enqueued`, `adaptation.retag_deduped`, `adaptation.retag_processed`, `adaptation.retag_failed`, `adaptation.low_similarity_captured`, `adaptation.retag_attempts_exhausted` <!-- audit-added -->, `adaptation.drain_summary` <!-- audit-added -->, `adaptation.capture_error` <!-- audit-added -->, `adaptation.retrieval_outcome_shape_unknown` <!-- audit-added -->.
