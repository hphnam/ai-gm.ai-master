<objective>
## Goal
Ship a `SuggestionService` that surfaces **non-persistent proactive suggestions** at two trigger points — `onConversationOpen(venueId)` and `onTurn(venueId, userMessage, conversationId?)` — composed from live mock-ops data via the existing `ToolDispatcher` (exported by `ChatModule` from 04-01). Deterministic-only for this plan: no Claude round-trip, no schema change, no persistence to `chat_messages`.

## Purpose
Phase 4 Goal from ROADMAP: "Proactive suggestions (below-par alerts, supplier cutoffs) from mock ops data." The chat loop (04-01) answers what the user asks. Proactive suggestions surface what the user **should** be asked to decide — the "24/7 GM" value proposition in PROJECT.md. This plan delivers the backend; 05-01 renders them.

## Output
- `ProactiveSuggestion` type in `@gm-ai/types` (published via dist).
- `SuggestionsModule` + `SuggestionsService` with two entry points, both returning `ProactiveSuggestion[]`.
- `probe-suggestions` end-to-end live probe (seeded Crown venue assertions).
- `.paul/phases/04-chat-engine/04-02-SUMMARY.md` (on UNIFY).
</objective>

## AC-2: `SuggestionsService.onConversationOpen(venueId)` returns deterministic suggestions
```gherkin
Given the seeded Crown venue has 5+ below-par stock items and 1+ supplier within the cutoff window
When `suggestionsService.onConversationOpen(CROWN_VENUE_ID)` is called
Then it returns an array where:
  - At least one suggestion has kind='below-par' with itemIds populated from get_stock_below_par results
  - At least one suggestion has kind='cutoff' when get_upcoming_cutoffs returns ok
  - Each suggestion has a human-readable `text` field (<= 160 chars) that does not leak raw UUIDs
  - `severity` is 'info' | 'warn' (warn for below-par items where currentQty === 0 or cutoff < 6h)
  - `sourceToolCall.tool` matches the tool_name used; `sourceToolCall.input` is the exact input object passed to the dispatcher
  - `generatedAt` is an ISO-8601 timestamp <!-- audit-added --> computed ONCE at the top of the method and shared by every suggestion emitted in the same call (internal consistency)
  - <!-- audit-added --> `text` has all control characters stripped (`\r`, `\n`, `\t` → space), has been trimmed, and is truncated at a word boundary (not mid-word) if composed length exceeds 160 chars
And when the dispatcher returns fail('no-data') for a branch, that branch contributes zero suggestions (silent, not an error)
And when venueId is malformed (fails UUID regex), the method returns [] (fail-soft, no throw)
And <!-- audit-added --> each dispatcher call is bounded by a 3000ms timeout via `Promise.race`; on timeout the branch contributes zero suggestions and an `error`-level log `{ event: 'suggestions.tool_timeout', venueId, tool, timeoutMs: 3000 }` is emitted
And <!-- audit-added --> when BOTH tools return fail('error') (dispatcher exception or timeout, NOT fail('no-data')), an `error`-level log `{ event: 'suggestions.both_tools_errored', venueId, trigger, details: [{tool, reason, detail}] }` is emitted so ops can detect service degradation
And a `suggestions.generate` metadata-only log is emitted with { venueId, trigger: 'conversation_open', suggestionCount, toolsInvoked, toolsFailed, latency_ms } — <!-- audit-added --> userMessage content, if any, is NEVER included in any log payload from this service
```

## AC-3: `SuggestionsService.onTurn(venueId, userMessage, conversationId?)` gates by lightweight heuristic
```gherkin
Given a user message
When `onTurn` is called
Then it runs the below-par check only if userMessage matches /stock|order|supplier|par|running low|out of/i (case-insensitive)
  And it runs the cutoff check only if userMessage matches /order|cutoff|deliver|supplier/i
  And if neither heuristic fires, it returns [] without invoking any tool
  And if both fire, it de-duplicates suggestions by (kind, itemIds[0] ?? tool)
  And it emits `suggestions.generate` log with trigger='turn'
And <!-- audit-added --> when conversationId is provided, the service performs a cross-tenant preflight: `prisma.chatConversation.findUnique({ where: { id: conversationId } })`; if the row does not exist OR its venueId !== venueId arg, the method returns [] and emits `{ event: 'suggestions.conversation_mismatch', venueId, conversationId }` at warn level. No silent correlation of another tenant's conversationId into this venue's log.
And <!-- audit-added --> the `suggestions.generate` log payload for trigger='turn' includes ONLY: { venueId, conversationId, trigger, messageLength, stock_matched: boolean, cutoff_matched: boolean, suggestionCount, toolsInvoked, toolsFailed, latency_ms }. userMessage content is NEVER logged — neither as raw text nor as hash nor as slice.
And empty/whitespace userMessage returns [] without invoking any tool
And <!-- audit-added --> the same 3000ms per-dispatch timeout and both-tools-errored escalation rules from AC-2 apply here as well
```

## SCOPE LIMITS
- **No Claude round-trip.** Suggestions are deterministic this plan. Claude-generated nuance (rewriting "5 below-par items" into natural "looks like you're low on X and Y — want me to draft the Heineken order?") is deferred to 04-03 or 05-01 as a downstream concern.
- **No persistence.** Suggestions are returned in-memory and re-derived on next call. A future "dismissed suggestions" feature is explicitly out of scope.
- **No HTTP controller.** Plan 05-01 exposes the REST surface; 04-02 stops at the service boundary — same shape as 04-01.
- **No onTurn Claude-adjudicated path.** The heuristic gate is a simple regex. If it misclassifies, 04-03's adaptation loop is the designed place to learn.
- <!-- audit-added --> **venueId remains trusted input; conversationId is NOT.** The controller layer (05-01) owns `venueId` tenant isolation (same boundary as MockOpsService). But when `conversationId` is provided to `onTurn`, the service MUST verify it belongs to `venueId` via a read-only Prisma lookup — without this check the service's own observability log would correlate another venue's conversationId with this venue's suggestion output, which is a log-integrity failure.
- **No retry on dispatcher `fail('error')`.** Log it and move on. This plan should not reinvent resilience; the dispatcher's structured error log is sufficient forensic signal. <!-- audit-added --> On timeout (3000ms per dispatch), the same rule applies — one timeout log, one `fail('error', 'timeout')` synthesised; no retry.
- **No new npm dep.** This plan must not add any package — every import resolves via existing workspace + apps/api deps.
- <!-- audit-added --> **No rate limiting in this plan.** The POC will not expose `/suggestions` publicly until Plan 05-01 adds a controller. When 05-01 adds the endpoint it MUST also add a `@nestjs/throttler` or equivalent guard — proactively tracked in Deferred Issues. If any consumer other than 05-01 starts calling this service at >1 Hz per venue, revisit.
- <!-- audit-added --> **No suggestion-count cap at the service layer.** Service returns up to 3 below-par + 2 cutoff = 5 suggestions per call. UI (Plan 05-01) owns visual pagination. If a future use case wants `limit`, extend the service signature — do not enforce in this plan.
- <!-- audit-added --> **No free-form DB text in observability payloads.** `suggestions.generate` and related events must carry ONLY structural metadata (tool names, counts, booleans, latency, venueId, conversationId). Supplier names, product names, notes, and userMessage never enter any log payload emitted from this service.
