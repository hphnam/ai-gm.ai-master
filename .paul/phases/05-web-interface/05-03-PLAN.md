<objective>
## Goal
Ship a read-only debug / observability route on apps/web that renders every provenance field Phase 4 already persists — per-message retrieval scores, tool inputs/outputs, feedback state, and re-tag queue status — correlated by conversationId so an operator can trace a single user turn from input through retrieval through Claude's tool loop through feedback and on into the adaptation queue.

## Purpose
The adaptation loop (04-03) and proactive suggestions (04-02) generate signal that's persisted to the database but currently invisible outside the seed scripts and probe harnesses. Without a debug surface, tuning retrieval thresholds, diagnosing thumbs-down reasons, or verifying that a low-similarity turn actually enqueued a re-tag item requires raw SQL — which defeats the "GM-available-24/7" POC demo story. 05-03 makes Phase 3/4 provenance legible so we can demo the *system learning* on the same call where we demo the *chat answering*.

## Output
- `apps/api` gains a `/debug/*` read-only surface (3 endpoints) exposing already-persisted fields — no new writes, no new side effects, no live retrieval re-runs
- `apps/web` gains a `/debug?venue=<uuid>&conv=<uuid>` route rendering those endpoints as a conversation inspector + re-tag queue panel
- `@gm-ai/types/debug` publishes the new response DTOs so UI imports them typed
- `probe-api.ts` grows 4+ assertions covering debug endpoints (happy path + cross-tenant + 404 + shape)
- Zero code modified in Phase 4 services or 05-01/05-02 endpoints
</objective>

## SCOPE LIMITS
- **Read-only** — no write endpoints, no "retry this item" / "clear failed" / "force drain" actions. Re-tag drain stays owned by AdaptationService.processReTagQueue(); debug only observes.
- **No live retrieval re-run** — scores are persisted in `toolCallLog.result.data[].similarity`; rendering them costs nothing. Re-running find() from debug would burn Voyage credit and introduce read-then-write races against the corpus.
- **No log aggregation UI** — http.request / chat.claude_call / retrieval.call / adaptation.* log events are stdout-only. A log tail UI needs OTel + a sink + a query endpoint; that's post-POC infrastructure.
- **No X-Request-Id correlation BEYOND the current call** — the debug UI shows its OWN most-recent successful requestId via `<DebugRequestIdBadge>` for forensic copy-paste, but correlating historical chat requestIds back to log entries requires the log aggregation described above. For now: operators use browser devtools Network tab + grep the api stdout.
- **No authentication / tenant gating beyond venueId** — same POC rule. Any bookmark to `/debug?venue=<uuid>` works for anyone who has the URL. Post-auth, debug routes go behind an operator-role guard. Defence-in-depth IS shipped: `X-Robots-Tag: noindex, nofollow` + Next.js `robots: { index: false, follow: false }` metadata prevent search-engine indexing; the amber warning banner discourages accidental screenshots; noindex is **NOT** a replacement for auth, just the best pre-auth defensive layer.   <!-- audit-added: must-have #4 documented -->
- **Debug surface is a point-in-time snapshot, not live** — neither debug hook refetches on focus / reconnect / interval. Operators explicitly refresh via the header button when they want fresh data. This makes debug traces reproducible and prevents observer-effect masking of real issues.   <!-- audit-added: must-have #2 -->
- **90-day retention window on conversation + message endpoints** — older data 404s at the debug layer. Raising the window OR exposing older data requires direct DB access (audit-logged by the DBA role). Retention is a typed constant `RETENTION_90D_MS` in `@gm-ai/types` — one-place-tunable for future deployment config.   <!-- audit-added: strongly-rec #8 -->
- **Tenant isolation is strict, not permissive** — retag-queue scoping uses `sourceMessage.conversation.venueId` OR venue-owned knowledge items; NEVER the permissive "include all globals" form that would leak cross-tenant activity on shared knowledge items. Probe D5 guards this.   <!-- audit-added: must-have #1 -->
- **Per-request + per-entry content caps** — server side: every `toolCallLog.result.data[].content` capped at 2048 chars with `__truncated: true` marker. Client side: `DebugJsonViewer` caps serialized payload at 65536 chars with a visible truncation banner. Raw full content is available via direct DB access for the <1% of operator investigations that need it.   <!-- audit-added: must-have #3 -->
- **No write-to-adaptation from debug UI** — creating test thumbs-down / manually enqueuing a retag item / flushing feedback are all out of scope. Those flows exist through /feedback (05-01).
- **No charts / time-series / histograms** — POC surfaces raw rows + counts only. Charting requires a chart lib dep + time-bucket API; deferred.
- **No conversation-list UI** — URL is still the persistence surface. Operators either come from the chat page (same ?conv=) or paste a conversationId directly. A conversation-list endpoint is post-auth.
- **No embedding / embeddingText rendering** — the api never selects these columns (Task 1 grep-verified); the UI never references these field names. Re-stating as a boundary so future debug expansion doesn't regress.
- **No dark mode / i18n / analytics / service-worker** — same post-POC scope as 05-02.
- **No virtualization** — operator traces are typically <50 messages + <200 queue items; raw DOM is fine at POC scale.
- **No new shared state store (zustand/jotai)** — React Query cache + URL params cover it, same as 05-02.
- **No @radix-ui/react-tabs dep** — initial draft included Tabs; audit removed it because the two-pane layout is a simple grid. Adding it later (if the debug page grows to genuinely need tabbed views) is a separate plan.   <!-- audit-added: must-have #6 -->
- **Radix version drift** — `@radix-ui/react-collapsible` pinned to `"latest"` per project convention. Post-POC: pin to exact majors in a tooling plan.   <!-- audit-added: can-defer -->
