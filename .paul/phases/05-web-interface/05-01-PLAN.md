<objective>
## Goal
Expose Phase 4 chat, suggestions, and adaptation capabilities as validated HTTP endpoints so the Plan 05-02 web UI can consume them without re-implementing service logic.

## Purpose
Phase 4 shipped three services (`ChatService`, `SuggestionsService`, `AdaptationService`) that are only reachable through probe scripts today. The web UI needs a thin, well-typed REST surface with input validation, cross-tenant preflight, and predictable error shapes — probe-testable via `curl`/`fetch` before any UI work begins.

## Output
- `VenuesController` (GET /venues)
- `ChatController` (POST /chat/messages, GET /chat/conversations/:id)
- `SuggestionsController` (POST /suggestions/on-open, POST /suggestions/on-turn)
- `FeedbackController` (POST /feedback)
- `ZodValidationPipe` shared across controllers
- `packages/types/src/api.ts` with request/response contracts re-exported for UI consumption
- `probe-api.ts` end-to-end HTTP verification script exposed as `pnpm --filter api probe:api`
</objective>

## AC-1: Venues list endpoint

```gherkin
Given the database has been seeded (2 venues: The Crown, The Anchor Bar)
When the client calls GET /venues
Then the response is 200 with an array of { id, name, city, channels } objects
And The Crown and The Anchor Bar are both present
And ids are UUIDs matching the seeded deterministic values
```

## AC-2: Chat send-message endpoint persists a full turn

```gherkin
Given a seeded venueId for The Crown and no existing conversationId
When the client POSTs { venueId, userMessage: "what needs ordering?" } to /chat/messages
Then the response is 200 with { conversationId, assistantMessage: { id, content }, toolCallLog, retrievedItemIds }
And conversationId is a freshly-created UUID
And assistantMessage.content is a non-empty string
And chat_messages contains exactly two rows for conversationId (user + assistant) with toolCallLog persisted on the assistant row
And the second POST reusing the returned conversationId appends a third user + fourth assistant message, preserving order via createdAt asc tiebreak id asc
```
