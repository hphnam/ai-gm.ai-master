<objective>
## Goal
Ship a working Next.js chat UI that lets a user pick a venue, hold a multi-turn conversation against the Plan 05-01 API, see proactive suggestions, and leave thumbs-up/down feedback — all with conversation persistence across refresh.

## Purpose
Plan 05-01 published the REST contract; 05-02 makes it usable. This is the first time a non-developer can actually *talk* to the GM assistant. Everything required by the POC success metrics in PROJECT.md ("stock queries return accurate data", "SOP retrieval returns relevant docs", "multi-turn context maintained", "thumbs feedback wired to adaptation loop") becomes demonstrable on this plan's completion.

## Output
- `apps/web` running on :3000 with working `/chat?venue=<uuid>&conv=<uuid>` route
- Venue selector (populated from `GET /venues`)
- Chat thread with optimistic user-message rendering + streaming-free assistant replies
- Suggestions surface (on-open + per-turn) rendered as cards above the composer
- Thumbs feedback buttons on every assistant message wired to `POST /feedback`
- All error paths surface ApiErrorResponse via toast (Sonner)
- Conversation persistence via URL search params (no browser storage abuse)
</objective>

## AC-1: Venue selector renders + persists to URL

```gherkin
Given the user opens http://localhost:3000/ with no ?venue= search param
When the page loads
Then they are redirected to /chat (or rendered a venue picker if /chat has no ?venue=)

Given the user is on /chat with no ?venue=
When they click the venue selector dropdown
Then they see The Crown + The Anchor Bar populated from GET /venues
And selecting a venue navigates to /chat?venue=<uuid>
And the selected venue label remains visible in a persistent header across refreshes
```

## AC-2: New conversation empty state

```gherkin
Given /chat?venue=<Crown-UUID> with no ?conv= param
When the page loads
Then the composer is visible and the thread area shows an empty state
   ("Ask anything about stock, ordering, SOPs, or suppliers.")
And the composer textarea auto-focuses on mount
And suggestions from POST /suggestions/on-open appear above the composer
   (0..N cards; if 0, no banner — do NOT render "no suggestions" placeholder)
```

## AC-3: Send message + optimistic render + conversation persistence

```gherkin
Given /chat?venue=<Crown-UUID>
When the user types "what needs ordering?" and submits
Then the UI immediately renders the user's message optimistically (role=user bubble, pending state marker)
And the composer clears + remains focused
And the submit button disables while pending (double-submit guard)
And on 200, the assistant reply renders below (role=assistant bubble)
And the URL updates to /chat?venue=<Crown-UUID>&conv=<returned-conversationId>
And POST /suggestions/on-turn fires in parallel with the send (Promise.all; UI renders suggestions when they arrive without blocking the chat reply)

Given /chat?venue=<Crown-UUID>&conv=<existing-conversationId>
When the page loads
Then GET /chat/conversations/:id?venueId=<Crown-UUID> fires on mount
And the full message history renders user→assistant→user→assistant in order
```

## AC-4: Thumbs feedback on assistant messages

```gherkin
Given an assistant message in the thread
When the user clicks the thumbs-up button
Then POST /feedback fires with { messageId, kind: 'up' }
And the button enters a selected state (highlighted)
And a success toast appears ("Thanks for the feedback")

Given the user changes their mind and clicks thumbs-down on the same message
When the request completes
Then POST /feedback fires with { messageId, kind: 'down' }
And the thumbs-up deselects; thumbs-down highlights
And the UI does NOT require a page reload to reflect the new state

Given the feedback endpoint returns 500
When the request fails
Then an error toast surfaces ("Feedback failed — please retry")
And neither button enters a selected state (optimistic rollback)
```

## AC-5: Error rendering conforms to ApiErrorResponse

```gherkin
Given any API call returns a 4xx or 5xx response with body { error: ApiErrorCode, details? }
When the response is handled by apiFetch()
Then it throws an `ApiError` instance carrying { status, error, details }
And the UI (via error boundary + React Query onError hook) surfaces a toast
   with a user-friendly mapping of error code to message

Given the API returns HTTP 413 (body too large)
When the composer submits oversized input (which shouldn't happen under schema max 8000 chars)
Then the user sees a toast "Message too long — please shorten and retry"

Given POST /chat/messages returns 400 with error=invalid-input
When the composer submits whitespace-only content
Then schema validation on the client (react-hook-form + zod) catches it BEFORE the network call fires
And a field-level error ("Message required") renders under the textarea
```

## AC-8: Assistant content rendered as plain text (XSS defence)  <!-- audit-added: must-have -->

```gherkin
Given an assistant message containing markdown tokens ("**bold**", "# heading", "<script>alert(1)</script>", multi-line content with \n)
When the message renders in a ChatMessage bubble
Then the raw characters appear verbatim — no HTML interpreted, no markdown parsed
And the bubble uses CSS `white-space: pre-wrap` so line breaks are visually preserved
And `dangerouslySetInnerHTML` is NEVER used in any component under apps/web/src
And no markdown renderer (`react-markdown`, `marked`, `markdown-it`, etc.) is added as a dependency
   — if the UX later requires rich rendering, a separate plan must land sanitization first
```

## AC-9: Accessibility baseline (WCAG AA minimums)  <!-- audit-added: must-have -->

```gherkin
Given the chat thread renders messages
When a screen reader navigates the page
Then the thread container has `role="log"` + `aria-live="polite"` so new messages are announced
And each icon-only feedback button has an `aria-label` ("thumbs up" / "thumbs down") + visible focus ring
And the composer textarea has a visible label (or `aria-label="Message"`) and an `aria-describedby` pointing at the submission hint
And severity on suggestion cards is NEVER conveyed by color alone — the card shows a text label ("Warning" / "Info") and/or a lucide icon (`AlertTriangle` / `Info`)
```

## AC-10: Observability correlation — X-Request-Id passthrough  <!-- audit-added: must-have -->

```gherkin
Given apps/api's 05-01 middleware echoes inbound X-Request-Id and generates UUID when missing
When apps/web makes any API call via apiFetch()
Then apiFetch generates a fresh UUID v4 per request and attaches it as `X-Request-Id` header
And the browser Network tab shows a different X-Request-Id per call
And a developer can grep the api `http.request` log lines by requestId to find the exact server-side trace for any UI action
And apiFetch reads the response `X-Request-Id` header and attaches it to ApiError instances for support/debug display
```

```gherkin
And all 4 messages re-render in order

Given a user bookmarks /chat?venue=<Crown>&conv=<uuid> and opens it fresh

```

- `packages/types/src/api.ts` — already published + consumed by 05-01. UI imports only; never mutates.

- `apps/api/src/main.ts` — CORS allowlist is set for http://localhost:3000; any other origin change is a separate plan.

## SCOPE LIMITS
- **No authentication / CSRF** — POC per PROJECT.md constraints. Future auth plan will wrap the chat page in a guard.
- **No streaming chat (SSE/WebSocket)** — 05-01 explicitly deferred streaming; responses appear on completion.
- **No debug panel** — that's Plan 05-03. No retrieval scores, no toolCallLog display, no re-tag queue visibility.
- **No conversation-list sidebar** — URL is the only persistence surface. Bookmarking a `?conv=` URL is the "resume" UX.
- **No localStorage / IndexedDB persistence** — URL params are sufficient; avoids privacy/clear-cache edge cases.
- **No `regenerate` feedback kind in the UI** — thumbs up/down only for this plan. Regenerate requires triggering a new Claude call which is a separate scope item.
- **No test suite / Playwright / Vitest** — manual human-verify is sufficient for POC UI; automated UI tests deferred.
- **No internationalization** — strings are English; externalize in a later plan if needed.
- **No dark mode toggle** — shadcn vars support it but the UI renders light-only in this plan.
- **No observability beyond what 05-01 already logs** — no client-side Sentry / PostHog / analytics in POC.
- **No Next.js 16 Server Actions** — use API routes via apiFetch; Server Actions would split the trust boundary across two ports.
- **(audit-added)** **No markdown rendering in assistant messages** — plain text with `whitespace-pre-wrap` only. If Claude emits markdown (`**bold**`, `# heading`), users will see the raw characters. Upgrading to sanitized markdown requires a separate plan that pins `react-markdown` + `rehype-sanitize` and defines an allowlist of tags — the XSS threat model here is too big to hand-roll.
- **(audit-added)** **No CSP headers on apps/web** — deferred until hosted deployment; in dev `http://localhost:*` blurs CSP. Add CSP via `next.config.ts` headers when deploying to Coolify.
- **(audit-added)** **URL-as-state privacy caveat** — `?conv=<uuid>` in URL = conversation-read access (combined with `?venue=<uuid>`). URL leaks to browser history, analytics referrers, shared-tab screenshots, etc. Acceptable at POC scope (no auth); post-auth, conversation IDs should move off URL and be fetched via a `/chat/conversations` list endpoint scoped to the authenticated user.
- **(audit-added)** **No virtualization on long threads** — <100 messages is fine as plain DOM; revisit if conversations grow past 100 turns.
