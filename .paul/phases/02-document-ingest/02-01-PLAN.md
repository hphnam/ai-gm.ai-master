<objective>
## Goal
Close the documented cross-organisation leak on `KnowledgeItem` by adding a mandatory `organizationId` FK, backfilling existing rows, and replacing the "venue-join OR (venueId IS NULL)" scoping hack across the docs service, the retrieval service, and all downstream callers with direct `organizationId` filtering.

## Purpose
Phase 1 established the multi-tenant trust boundary (auth, organizations, role guards). The existing `KnowledgeItem` row shape — `venueId String?` with org derivation through `venue.organizationId` — has a documented gap: rows with `venueId = NULL` ("global" docs) have no organisation ownership at all, so `docs.service.ts` explicitly acknowledges "Global docs (venueId null) are shared across all orgs until per-org ownership lands in a later phase" (docs.service.ts:42). The retrieval SQL repeats the same hack (`"venueId" = $2 OR "venueId" IS NULL`), so any global doc from any tenant can be retrieved by any other tenant's chat session. This is a SOC-2 CC6.6 failure equivalent to the retag-queue leak closed in Plan 05-03's audit. Phase 2's headline feature (manager-uploaded documents) cannot land without this fix — uploading an internal SOP to one org and having another tenant's chat retrieve it is the worst possible first impression.

## Output
- `KnowledgeItem.organizationId` column + FK + index on schema.
- Idempotent backfill migration populating existing rows from `venue.organizationId` (venue-scoped rows) or the Demo Organization (orphan/null-venue rows, scoped to the dev seed corpus).
- `docs.service` list/getById/create paths scoped by `organizationId` directly — venue-join OR-clause removed.
- `RetrievalService.find()` signature takes a required `orgId`; SQL gains `"organizationId" = $N`; both SQL branches updated.
- `ToolDispatcher` threads `ctx.orgId` into every `find_knowledge` call.
- `IngestService.ingest()` writes `organizationId`; `docs.service.create` + seeder + `save_knowledge_doc` tool all supply it.
- `probe-api` gains ≥3 new assertions proving cross-org isolation: global-doc list leak closed, getById leak closed, retrieval cross-org leak closed.
- `apps/api/src/scripts/seed.ts` writes `organizationId` on every `KnowledgeItem` insert.
</objective>
