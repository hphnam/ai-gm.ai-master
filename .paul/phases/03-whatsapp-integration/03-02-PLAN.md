## Goal
Ship `probe-whatsapp.ts` that hermetically exercises every Plan 03-01 acceptance criterion (AC-1 through AC-12) against a bootstrapped NestApplication, plus AC-10 timeout simulation via a dev-only `PROBE_CHAT_SERVICE_DELAY_MS` env injected into `ChatService.sendMessage`. ## Purpose
Plan 03-01 shipped the WhatsApp webhook surface but left all curl-loop assertions (AC-3 two-hit rate limit, AC-8 replay, AC-9 30/h throttle, AC-10 timeout) as manual smoke. A green probe closes the audit-defensibility gap — every AC becomes regression-tested on every run. Also exercises the M1 URL-pin, M2 dev-bypass, M7 malformed-params branches end-to-end. Audit-added: dev-bypass positive-path coverage (03-01 AC-12 accept branch); no-double-reply safety after timeout; PROBE_CHAT_SERVICE_STUB mode for CI-affordable runs.

## Output

- `apps/api/package.json` — `"probe:whatsapp": "tsx src/scripts/probe-whatsapp.ts"` npm script.

## Claude-spend profile (audit-added M1)
A default probe run makes **~34 real Claude API calls** through ChatService (W8:1 + W9:1 + W10:1 + W12:30 + W13:1-orphan). At ~$0.01-0.03 per call, a single probe run costs **~$0.30-1.00**. Two paths:
- **Default (stub-off):** real Claude path exercised end-to-end. Use for release-gate manual runs.
- **Stubbed (`PROBE_CHAT_SERVICE_STUB=true`):** ChatService skips Claude, returns a deterministic `assistantMessage` fixture. Use for CI / frequent dev runs. **Zero** external API cost.
Both modes PASS the assertion set — the stub returns a fixed body that still flows through the adapter and gets persisted. Real-Claude mode is recommended before tagging a release.
