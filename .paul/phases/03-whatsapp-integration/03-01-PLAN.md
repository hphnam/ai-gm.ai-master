<objective>
## Goal
Land the WhatsApp inbound webhook end-to-end for text messages only: POST `/webhooks/twilio/whatsapp` validates Twilio's HMAC-SHA1 signature against a fixed public URL, resolves the sender's phone to a verified User → Organization → default Venue, calls `ChatService.sendMessage()` under a hard timeout, and replies via `WhatsAppAdapter.sendText()`. Non-text inbound (image/audio) gets a placeholder rejection reply. Unknown senders get a rate-limited onboarding reply. Verified senders are rate-limited to cap Claude cost exposure.

## Purpose
Phase 3 payoff — proves the WhatsApp channel connects into the existing v0.1 + v0.2 AI loop (Auth + Phone linking + ChatService) without polluting the existing probe/test surface. Adapter + controller + routing ship this plan; the hermetic `probe-whatsapp.ts` + real-sandbox UAT split out to 03-02.

## Output
- `WhatsAppModule` at `apps/api/src/modules/whatsapp/` (adapter + controller + service + signature guard + two in-memory rate limiters + seen-sid dedupe)
- `TwilioWebhookPayloadSchema` + outbound-result + cost-ceiling constants in `@gm-ai/types`
- Env keys (wired through `assertAuthEnv`): `TWILIO_WHATSAPP_FROM`, `TWILIO_WHATSAPP_DRIVER_OVERRIDE`, `WHATSAPP_WEBHOOK_PUBLIC_URL` (audit-added M1), `ALLOW_WEBHOOK_DEV_BYPASS` (audit-added M2)
- `/webhooks/twilio/whatsapp` path-filtered for `urlencoded` body parsing in `main.ts` (Twilio sends `application/x-www-form-urlencoded`, not JSON)
- App module registers `WhatsappModule`
- `.env.example` documents new env keys

No probe scripts in this plan (probe-whatsapp.ts ships in 03-02). Verification is TypeScript build + manual HMAC-signed curl + existing probe-api + probe-auth regression parity.
</objective>

## AC-1: Verified sender + text → AI reply
```gherkin
Given the POST body has a valid X-Twilio-Signature header computed against WHATSAPP_WEBHOOK_PUBLIC_URL + sorted form params with TWILIO_AUTH_TOKEN
  And `From` resolves to a User with phoneVerifiedAt != null, an OrganizationMember row, and at least one Venue in that Organization
  And `Body` contains text and NumMedia = "0"
  And the sender is under the per-hour inbound rate-limit ceiling
When Twilio POSTs to /webhooks/twilio/whatsapp
Then the server responds 200 within 14 seconds (Twilio's retry budget is 15s)
  And ChatService.sendMessage is invoked with resolved orgId, userId, venueId, and Body as userMessage, wrapped in a 12s Promise.race timeout
  And WhatsAppAdapter.sendText is invoked with the assistant reply text and the sender's From number
  And a whatsapp.inbound structured log line is emitted with { from: sha256Prefix, messageSid, waIdHash, bodyLength }
  And a whatsapp.outbound structured log line is emitted with { to: sha256Prefix, mode, latencyMs }
```

## AC-2: Invalid HMAC signature → 403
```gherkin
Given the request body is well-formed
  And X-Twilio-Signature header is missing, malformed, or does not validate against WHATSAPP_WEBHOOK_PUBLIC_URL + sorted form params
When Twilio (or an attacker) POSTs to /webhooks/twilio/whatsapp
Then the server responds 403 with body `{"error":"signature-invalid"}`
  And no ChatService, WhatsAppAdapter, or phone-lookup call is made
  And a whatsapp.signature_rejected structured log line is emitted with { reason, signaturePresent: boolean }
  And no raw URL query-string or body content appears in the log line
```

    **apps/api/src/modules/whatsapp/whatsapp-signature.guard.ts (new):**
    - `@Injectable()` class `WhatsappSignatureGuard implements CanActivate`.
    - **Signature computation (audit-added M1: URL-pinned, M7: deterministic body):**
      1. `const publicUrl = assertAuthEnv().whatsapp?.webhookPublicUrl` — NEVER derive URL from req headers. If missing at request-time (dev mode where whatsapp env isn't populated), reject as signature-invalid.
      2. Take `req.body` (Express urlencoded parser has already parsed it). **Reject if any value is an array / non-string** (Twilio sends only scalar strings; duplicate-key payloads from attackers produce arrays via qs — reject these as signature-invalid with reason `malformed-params`).
      3. Sort keys via `Object.keys(body).sort()` (lexicographic, case-sensitive — matches Twilio's spec).
      4. Concatenate `publicUrl + sortedKeys.map(k => k + body[k]).join('')` → signing string.
      5. HMAC-SHA1 with `process.env.TWILIO_AUTH_TOKEN` as key, base64-encode digest → expected.
      6. Constant-time compare to `X-Twilio-Signature` header via `crypto.timingSafeEqual` on equal-length Buffers (lengths mismatched → early false).
    - **Dev-bypass (audit-added M2):**
      - Read `assertAuthEnv().whatsapp?.allowDevBypass` (already `false` in prod thanks to M2 boot guard).
      - If `allowDevBypass === true` AND `process.env.TWILIO_AUTH_TOKEN` is unset AND header `X-Twilio-Signature` === `'probe-console'`: accept the request. Emit `whatsapp.signature_dev_bypass` log at WARN level WITH `{ path, allowDevBypass: true, nodeEnv: process.env.NODE_ENV }` **on every acceptance** (not just boot — operators need a persistent WARN trail if dev-bypass is active). If any of those conditions fail, the request follows the normal signature path.
      - D-03-01-F (pre-Phase-4 go-live blocker): remove this entire dev-bypass branch + `ALLOW_WEBHOOK_DEV_BYPASS` env var before prod cutover. Add to SUMMARY alongside D-01-02-F + D-01-03-F.
    - **Failure logging:** `whatsapp.signature_rejected { reason, signaturePresent: boolean, bodyKeyCount }` — NEVER log body values, NEVER log the signature string, NEVER log the raw URL query string (the publicUrl is env-known so no leak there).
    - On failure: throw `new ForbiddenException({ error: 'signature-invalid' })`.
