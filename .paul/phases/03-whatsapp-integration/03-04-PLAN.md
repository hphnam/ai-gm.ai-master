<objective>
## Goal
Replace the Twilio WhatsApp provider with Infobip. Same functional surface as shipped by Plans 03-01/02/03 (inbound → ChatService, outbound via adapter, 2h idle conversation reuse, 24h proactive opener, image multimodal with SSRF/MIME/magic-byte hardening, audio rejection, verified-sender throttle, MessageSid-style replay dedupe). Different wire: HMAC-SHA256 over raw JSON body (vs Twilio's HMAC-SHA1 over URL+sorted-fields), JSON `{ results: [...] }` inbound payload (vs form-urlencoded), `POST /whatsapp/1/message/text` with `Authorization: App {API_KEY}` outbound (vs `POST /2010-04-01/Accounts/{sid}/Messages.json` with Basic Auth).

Delete the probe-whatsapp harness (no longer used for verification). Keep typing-indicator-timers + composeOpenerText + cross-tenant scoping + Phase 1/2 untouched.

## Purpose
Twilio Sandbox UAT pre-flight surfaced:
  - SK API Key vs AC Account SID confusion for inbound HMAC
  - Twilio WhatsApp sender purchase friction
  - Meta Business verification calendar-gate even for trial senders
  - Convoluted Console UX

Infobip is a tier-1 WhatsApp BSP, ships with a free trial sender (`447860088970` — Ryan's allocated trial number), and exposes a straightforward HMAC-SHA256 webhook + REST outbound API. This plan closes Phase 3 by delivering the same engineering outcome over a provider the team can actually ship against.

## Output
- Provider-neutral naming where practical: `whatsapp-media-download.ts` (renamed from `twilio-media-download.ts`), `WHATSAPP_MEDIA_HOST_ALLOWLIST` env (renamed from `TWILIO_MEDIA_HOST_ALLOWLIST`), `DEFAULT_WHATSAPP_MEDIA_HOST_ALLOWLIST` (renamed from `DEFAULT_TWILIO_MEDIA_HOST_ALLOWLIST`). Provider-specific naming on the wire-layer files only (adapter + signature guard).
- Infobip webhook path: `POST /webhooks/infobip/whatsapp` (renamed from `/webhooks/twilio/whatsapp` — route path change requires Infobip Portal webhook URL update on Ryan's side at UAT time).
- `InfobipInboundWebhookSchema` in `@gm-ai/types` replaces `TwilioWebhookPayloadSchema`.
- Infobip env block in `assertAuthEnv()` replaces the Twilio-WhatsApp block (Twilio Verify block for SMS OTP — Phase 1 Plan 01-03 — is UNTOUCHED; those are two separate Twilio integrations and only the WhatsApp one migrates).
- `probe-whatsapp.ts` deleted; `probe:whatsapp` npm script removed; Twilio-era `__resetForTest` exports in rate-limit/seen-sid/typing-timer modules left in place (harmless, single-line exports, removing them is scope creep).
- D-03-03-F closed. D-03-04-F registered: Infobip has no public typing-indicator endpoint for WhatsApp either — typing ships **console-mode only**, same fail-soft pattern as the Twilio version.

## AC-1: Infobip HMAC-SHA256 signature verifies → 200; invalid → 403
```gherkin
Given INFOBIP_WEBHOOK_SECRET is set and the webhook receives a POST to /webhooks/infobip/whatsapp
  And the request body is raw JSON
  And the signature header (name confirmed via Infobip docs at APPLY time — `X-Callback-Signature` or `X-Webhook-Signature`) is present
When the signature header value equals `hmacSha256Hex(rawBodyBytes, INFOBIP_WEBHOOK_SECRET)` (format confirmed at APPLY — hex vs base64 vs `sha256=<hex>` prefix)
Then the guard passes
  And handleInbound is invoked

Given the signature header is absent OR mismatches
When the guard runs
Then the response is 403 with `{ error: 'signature-invalid' }`
  And a `whatsapp.signature_rejected` warn log emits with { reason, signaturePresent, bodyLen } (NO raw body in logs)

Given the signature header VALUE is not parseable in the declared encoding (audit-added M3)
  e.g. odd-length hex string, non-hex chars when encoding=hex, non-base64 chars when encoding=base64
When the guard attempts Buffer.from parse
Then the guard rejects with `reason: 'signature-malformed'` BEFORE `timingSafeEqual` is reached
  (no control-flow relies on the equality check to catch malformed encoding)

Given the JSON body parse fails AFTER signature validation succeeded (audit-added S2 — G9)
When the controller logs `whatsapp.payload_invalid`
Then the log includes `signatureValidated: true`
  (compliance audit-trail distinguishes "signed by someone with the secret but malformed JSON" from "unsigned garbage")
```

      import { CanActivate, ExecutionContext, ForbiddenException, Injectable, Logger } from '@nestjs/common'
      import { assertAuthEnv } from '../auth/assert-auth-env'

      // Infobip webhook signature: HMAC-SHA256 over RAW request body bytes, keyed by INFOBIP_WEBHOOK_SECRET.
      // Header name + encoding: confirmed at APPLY via <URL>.
      const SIGNATURE_HEADER = 'x-callback-signature'   // adjust after WebFetch
      const ENCODING: 'hex' | 'base64' = 'hex'          // adjust after WebFetch

      @Injectable()
      export class WhatsappSignatureGuard implements CanActivate {
        private readonly logger = new Logger(WhatsappSignatureGuard.name)

        canActivate(ctx: ExecutionContext): boolean {
          const req = ctx.switchToHttp().getRequest()
          const sig = req?.headers?.[SIGNATURE_HEADER]
          const signaturePresent = typeof sig === 'string' && sig.length > 0
          const rawBody: Buffer | undefined = req?.rawBody
          const bodyLen = rawBody?.length ?? 0

          const env = assertAuthEnv()
          const secret = env.infobip?.webhookSecret
          const allowDevBypass = env.infobip?.allowDevBypass === true

          // Dev-bypass (03-01 pattern preserved, gated on NODE_ENV!=production via assertAuthEnv).
          if (allowDevBypass && !secret && signaturePresent && sig === 'probe-console') {
            this.logger.warn('whatsapp.signature_dev_bypass', { nodeEnv: process.env.NODE_ENV })
            return true
          }

          if (!secret) this.reject('config-missing', signaturePresent, bodyLen)
          if (!signaturePresent) this.reject('missing-signature', signaturePresent, bodyLen)
          if (!rawBody) this.reject('no-raw-body', signaturePresent, bodyLen)

          // Strip optional "sha256=" prefix if Infobip uses GitHub-style format.
          const received = (sig as string).startsWith('sha256=') ? (sig as string).slice(7) : (sig as string)
          const expected = createHmac('sha256', secret!).update(rawBody!).digest(ENCODING)

          const recBuf = Buffer.from(received, ENCODING === 'hex' ? 'hex' : 'base64')
          const expBuf = Buffer.from(expected, ENCODING === 'hex' ? 'hex' : 'base64')
          if (recBuf.length !== expBuf.length) this.reject('length-mismatch', signaturePresent, bodyLen)
          if (!timingSafeEqual(recBuf, expBuf)) this.reject('hmac-mismatch', signaturePresent, bodyLen)

          return true
        }

        private reject(reason: string, signaturePresent: boolean, bodyLen: number): never {
          this.logger.warn('whatsapp.signature_rejected', { reason, signaturePresent, bodyLen })
          throw new ForbiddenException({ error: 'signature-invalid' })
        }
      }
