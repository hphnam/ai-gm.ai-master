<objective>
## Goal
Re-swap the WhatsApp provider from Infobip → Twilio Conversations API, and reshape the new-user onboarding from in-band OTP-reply to a signed onboarding link sent via WhatsApp. Existing-user phone linkage stays web-initiated with a WhatsApp confirmation code entered on the web.

## Purpose
The Infobip integration (Plans 03-04, 03-05) shipped working but constrains the UX:
- No native typing indicator (D-03-04-F documents this — console-mode only).
- In-band OTP forces every new user through a 6-digit reply flow before any chat — high friction, and the OTP message is unstructured WhatsApp text.
- Phone verification rides on Infobip 2FA SMS as a separate product, which adds Portal setup and two cred surfaces.

Twilio Conversations API restores:
- Native typing indicators + read receipts on a documented endpoint.
- Persistent `Conversation` resources (one per phone) — cleaner audit trail and richer media handling.
- Single product for invite delivery, chat, and phone-verify codes (one auth-token surface, one webhook).

Signed-link onboarding matches modern SaaS patterns (Slack, Linear) and gives the manager a clear "invite sent" signal in WhatsApp delivery receipts.
</objective>

<decisions>
1. **Twilio Conversations API** (not Programmable Messaging) — chosen for native typing indicators + persistent conversation model.
2. **Hard swap** — Infobip is removed in the same phase. No `WHATSAPP_PROVIDER` flag. Rollback = revert the phase commits.
3. **New users**: WhatsApp DM with signed onboarding link → web `/onboard?token=` → name + better-auth signup → atomic redeem+link+session-create. The in-band OTP-reply flow (current `whatsapp-onboarding-state` invite_code branch) is dropped.
4. **Existing users**: web-initiated, WhatsApp 6-digit code, entered on web. Same data flow as today's `InfobipVerifyService`, renamed to `WhatsappVerifyService`, routed through the new Twilio adapter.
5. **Conversation SID storage**: new `WhatsappConversation` Prisma model maps `phoneNumber @unique → conversationSid`. Adapter does find-or-create; webhook can upsert on `onConversationAdded`.
6. **Webhook**: single Conversations Service-scoped webhook URL, `POST /webhooks/twilio/conversations`, `application/x-www-form-urlencoded`. Gate on `EventType == 'onMessageAdded'`; ignore others with 200.
7. **Signature**: Twilio's `X-Twilio-Signature` — HMAC-SHA1 base64 over `(URL + sorted form-field key+value concat)`. Auth-token doubles as signing secret.
</decisions>

## Task 3 — WhatsAppAdapter rewrite
Keep public surface (`sendText(to, body)`, `sendTypingIndicator(messageId)`) returning `WhatsAppOutboundResult`. Internally:
- `resolveConversationForPhone(phone) → conversationSid`:
  1. Lookup `WhatsappConversation` by `phoneNumber`.
  2. If missing: `POST /v1/Services/{ServiceSid}/Conversations` to create, then `POST /Conversations/{Sid}/Participants` with `messaging_binding.address=whatsapp:+E164` and `messaging_binding.proxy_address=TWILIO_WHATSAPP_SENDER`. Persist row.
  3. Catch `P2002` (race) by re-reading.
- `sendText`: write a message to the resolved conversation: `POST /Conversations/{Sid}/Messages` with `Author=system`, `Body=…`.
- `sendTypingIndicator`: `POST /v1/Services/{Sid}/Conversations/{ConvSid}/Webhooks` (Twilio's typing endpoint, see Conversations docs). Map `messageId` (caller passes inbound `MessageSid`) to the conversation via reverse-lookup; on miss, no-op.

Auth: Basic `account_sid:auth_token`. Timeout 10s, abort on signal.

## Task 4 — Webhook schema + parser
Replace `InfobipInboundWebhookSchema` with `TwilioConversationsEventSchema` matching `onMessageAdded` body shape (form-urlencoded → already coerced to flat keys by NestJS): `EventType, ConversationSid, MessageSid, Author, Body, Source, ParticipantSid, Media (array via NumMedia + MediaUrl{N}), DateCreated`.

Normalize into a `WhatsappInboundResult` (rename type) that downstream `whatsapp.service.ts` already consumes — the rename is the only churn for the service.

## Task 5 — Signature guard swap
`whatsapp-signature.guard.ts`: change from HMAC-SHA256-over-raw-JSON to Twilio's algorithm:
1. Read `X-Twilio-Signature` header (base64).
2. Build the signing string: `FULL_PUBLIC_URL` + concatenation of (`key + value`) pairs from the body sorted by key.
3. `HMAC-SHA1(TWILIO_AUTH_TOKEN, signingString)` → base64.
4. Constant-time compare.

Distinct rejection reasons preserved (config-missing, missing-signature, signature-malformed, hmac-mismatch).
Body parser for `/webhooks/twilio/conversations` must be urlencoded (configure in `main.ts`).

## Task 6 — Controller swap
Rename route to `POST /webhooks/twilio/conversations`. Drop the `results[]` batch loop (Twilio fires one event per request). Gate `EventType==='onMessageAdded'`; return 200 for everything else with debug log. `MessageSid` is the idempotency key (already wired through `markAndCheckSid`).

## Task 7 — Invite flow → signed link
`InviteService.create`:
- Still generates the existing 8-char Crockford code (it's a stable invite identifier, still useful for log correlation), but ALSO mints a signed token: `HMAC-SHA256(server_secret, inviteId + expiresAt)` encoded as `<inviteId>.<expHex>.<sigB64url>`.
- Sends WhatsApp message via adapter: `"You've been invited to <orgName> on GM AI by <issuerName>. Tap to set up your account: <PUBLIC_APP_URL>/onboard?t=<token>. Expires in 24h."`
- Manager UI no longer surfaces the code as primary; we display it as a fallback the manager can read aloud (rare path).

Web `/onboard` page:

- POST submits name (+ better-auth signup hook if no User row by email). Server-side handler does atomic redeem + phone-link + session create — same code path as today's `linkUserAndWelcome` in `whatsapp-onboarding.service.ts`, lifted into a reusable `redeemInvite(token, profile)` function.

State machine: collapse the `unknown` branch — drop `invite_code` intent recognition. Unknown numbers get a single friendly "Your number isn't linked — ask your manager to invite you" reply (rate-limited as today). `otp_pending` state can stay but is unreachable through invite flow; the phone-verify path uses it briefly only for unverified existing users.

<risks>
- **Twilio Conversations sandbox limits**: sandbox can only message numbers that have joined it. Production-ready WhatsApp Sender requires Meta Business approval — this is a process risk, not a code risk. Document in CLAUDE.md.
- **`PUBLIC_WEBHOOK_URL` mismatch**: Twilio signs the URL you configured in their console. If our deployed URL differs (behind a proxy, custom domain), signature validation breaks silently. Mitigation: read the public URL from env, not from `req.url`.
- **Conversation SID race**: two inbound webhooks for a new phone could race to create a Conversation. Mitigation: rely on Twilio's idempotency for create + DB `@unique` constraint, catch P2002.
- **In-flight invites at swap time**: any pending Infobip invite codes won't work post-swap. Mitigation: include a migration step that marks pending invites as `revoked` with a manager-visible reason ("provider migration — please re-invite"). Or accept the breakage if there are no pending invites — confirm with a count query before merge.
- **Signed-link token replay**: token is one-shot. Server marks invite redeemed atomically; second use returns the "already used" reply.
</risks>
