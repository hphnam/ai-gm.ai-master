<objective>
## Goal
Land the hierarchical schema (`KnowledgeSection` + `KnowledgeChunk`) and section detection at ingest. New uploads produce **doc → section → chunk** structure on the new tables; existing `KnowledgeItem` rows are untouched (backfill is 01-02). This is the foundation for v0.3's "neural network of data" feel — every subsequent piece (graph layer, graph-aware notifications, procedural runtime) depends on retrieval injecting whole containing sections, not flat chunks.

## Purpose
v0.3's headline value is the assistant feeling like a **secondary brain for the company** — connected, associative, "knows the venue." Section-level injection is what makes a vector hit feel like clicking into a note instead of reading a fragment. This plan makes that possible additively, without breaking the v0.2 retrieval surface that's currently in production.

## Output
- 2 new Prisma models (`KnowledgeSection`, `KnowledgeChunk`) + additive migration applied clean to NeonDB via `prisma migrate diff → deploy` (project convention)
- Zod contracts in `@gm-ai/types` for both models + `SectionDetectionResult`
- `SectionDetector` service module — extractor-first → regex fallback → cap-aware split → graceful chunk fallback for pathological cases
- `IngestService` extended to populate sections + chunks during the existing transactional upsert
- `probe-section.ts` with ≥10 assertions covering boundary detection, cap enforcement, cross-org isolation, idempotency
- `01-01-SUMMARY.md` with per-AC outcomes, deviations, deferred items
</objective>

## AC-3: Section detection — extractor-emitted markers honored
```gherkin
Given a freshly-ingested document with extractor-emitted structure
When `SectionDetector.detect(content, mimeHint)` runs
Then for each input shape it produces the expected section count and titles:
  - PPTX content (verified shape from 04-01: `## Slide N: <title>\n<body>\n[notes]\n## Slide N+1: ...`) → N sections, titles `Slide N: <title>`
  - DOCX content (verified shape from 04-01: mammoth produces plain text with heading lines preserved as standalone lines; detector treats heading-style lines (Title Case ≤80 chars, immediately followed by content) as boundaries) → 1 section per heading
  - Markdown with `## Heading A\n...\n## Heading B` → 2 sections, titles ["Heading A", "Heading B"]
  - **CSV with row-batch policy (audit-M4 — prevents 10K-row inventory exploding into 10K Voyage calls):** group every `CSV_ROW_BATCH_SIZE = 50` rows into one section; first row treated as header and prepended to each batch's content for retrieval clarity; section title = `Rows {start}-{end}`. A 10K-row CSV produces 200 sections, not 10K.
  - XLSX-like with `# Sheet: <name>\n...\n# Sheet: <other>` → 1 section per sheet; rows within a sheet that exceed sheet's soft cap fall through to AC-4 sub-heading/chunk-fallback policy (sheets are sub-heading-free, so they hit the truncated:true graceful-degrade path — verified safe by W9)
  - Plain-text PDF with no structural markers → falls through to heading regex (`splitByHeadings` per Task 2 step 1)
  - Document with no detectable structure → returns 1 single section spanning whole content with metadata sentinel `needsClustering: true` (deferred LLM-clustering trigger). **Cost guard (audit-M3):** if that single section's tokenCount > `MAX_EMBEDS_PER_DOCUMENT * CHUNK_TARGET_TOKENS` it triggers the per-doc embed cap rather than blasting Voyage uncapped — see AC-5.<!-- audit-added M4 + M3 -->
```

## AC-4: Section size cap — soft 4K, hard 8K, graceful degrade
```gherkin
Given a section with computed token count
When the cap policy applies
Then:
  - tokenCount ≤ 4096 (soft cap) → stored as 1 section, `truncated: false`
  - 4096 < tokenCount ≤ 8192 with sub-headings → split at sub-heading boundaries; each split is its own section; sectionIndex incremented; `truncated: false` on splits
  - tokenCount > 8192 with sub-headings → split at sub-heading until each split ≤ 4096; if any leaf still > 8192 → flat-chunk fallback applied to that leaf only
  - tokenCount > 4096 with no sub-headings to split on → flat-chunk fallback (sliding-window 1024-token chunks, 128-token overlap), section retains full content but `truncated: true`
  - **Recursion depth bound (audit-M5):** heading split is recursive (split → if leaf still over cap, split sub-headings). Bound `MAX_HEADING_RECURSION_DEPTH = 8`. Past depth, force flat-chunk fallback for that leaf regardless of remaining headings. Prevents pathological docs (deeply nested h1→h2→h3→...→h8) from stack-overflow.<!-- audit-added M5 -->
And token counting uses a deterministic counter (initial implementation: `Math.ceil(content.length / 4)` as a stable approximation; **non-Latin-aware caveat (audit-S3):** the 4-chars-per-token heuristic over-counts CJK by ~4x and under-counts emoji-dense text — JSDoc on `estimateTokens()` documents this explicitly; D-01-01-A trigger revised to "first non-Latin doc surfaces OR billing-grade counts needed").<!-- audit-added S3 -->
```

## AC-5: Chunk creation — every section gets ≥1 chunk; embeddings via $executeRaw with bounded concurrency, retry, timeout, and per-doc cost cap
```gherkin
Given a section persisted via the new ingest path
When chunk generation runs
Then:
  - section.tokenCount ≤ 1024 → 1 chunk containing the full section content
  - section.tokenCount > 1024 → multiple chunks, sliding-window (1024 tokens, 128 overlap), chunkIndex 0-based
  - **Two-phase persistence (audit-M1):** Phase 1 — Prisma transaction creates KnowledgeItem + KnowledgeSection + KnowledgeChunk rows with `embedding: null`, `embeddingText` populated. Transaction commits BEFORE any Voyage HTTP call. Phase 2 — post-commit, embeddings flow into chunks via `EmbeddingsService.embedDocument(chunk.embeddingText)` → `$executeRaw\`UPDATE knowledge_chunks SET embedding = ${vector}::vector WHERE id = ${chunkId}\`` (1024-dim verified per AC-8 W14). This eliminates Voyage-latency-inside-DB-transaction lock contention on large docs (50-page PDF would hold DB locks 50+ seconds under inline-transaction model).
  - **Bounded concurrency (audit-M2):** at most `MAX_CONCURRENT_CHUNK_EMBEDS = 3` parallel Voyage calls per ingest; queue overflow rejected with `EMBED_QUEUE_TIMEOUT_MS = 15_000` 15s wait — chunk left with embedding=null + `ingest.embed_queue_timeout` log. Mirrors 04-01 image-extractor semaphore pattern.
  - **Per-ingest overall timeout (audit-M2):** `INGEST_EMBED_PHASE_TIMEOUT_MS = 120_000` 2min hard cap on the entire post-commit embed phase via AbortController; any chunk not embedded by deadline left with embedding=null + `ingest.embed_phase_timeout` log. Mirrors 04-03 extractor pattern.
  - **Retry on transient (audit-M3):** Voyage 5xx and 429 responses retry once after 1s linear backoff; 4xx (other than 429) and post-retry 5xx fail-soft with `ingest.embed_failed` log carrying sanitised error reason. Mirrors 04-03 Claude retry pattern.
  - **Per-document embed budget (audit-M3):** `MAX_EMBEDS_PER_DOCUMENT = 200` (~50K tokens at 1024 chunk target). If section detection produces > 200 chunks for a single doc, embed only the first 200 in section/chunkIndex order; remaining chunks persist with embedding=null + `ingest.embed_cap_exceeded` log carrying chunkCount + cap. Prevents accidental or malicious 100-page PDF blasting 100K tokens through Voyage.
  - On any embed-fail path → chunk row exists with `embedding: null`, log emitted; ingest controller returns 200 to caller (the document is still searchable in 01-02 once retrieval falls back to KnowledgeItem.embedding for chunks where chunk.embedding IS NULL; documented in 01-02 scope).<!-- audit-added M1 + M2 + M3 -->
```
