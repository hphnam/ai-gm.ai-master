<objective>
## Goal
Backfill `KnowledgeSection` + `KnowledgeChunk` rows for every existing `KnowledgeItem` that doesn't yet have them (idempotent), and switch retrieval to inject the **best-matching section** content instead of the full `KnowledgeItem.content` for any KI that has been backfilled. Pre-backfill KIs continue to fall through to `ki.content` so retrieval never goes blank during rollout. After this plan, the chat path delivers the "I clicked into the note" feel that v0.3 Phase 1 promised.

## Purpose
01-01 landed the schema and ingest path additively — new uploads write doc → section → chunk, but existing rows still have zero section/chunk children and retrieval still pulls whole-document content. This plan closes the loop: every existing KI gets sections, and retrieval returns one focused section per hit. That changes the cost profile (smaller payloads → cache wins land in 01-03), the user experience (specific section quoted, not a wall of text), and unblocks the Phase 2 graph layer (DocLink edges resolve to section IDs, not document IDs).

## Output
- `apps/api/scripts/backfill-knowledge-sections.ts` — idempotent backfill walking unsectioned KIs, calling existing `IngestService.persistSectionsAndEmbed` (refactored to a public-callable surface). Per-tenant cost log; aggregate summary at end.
- `apps/api/package.json` — `backfill:sections` npm script.
- Refactored `retrieval.service.ts` — for `entityType='knowledge_item'` hits, expand to best-matching section via single LATERAL JOIN follow-up query; fall back to `ki.content` when no sections exist for that KI.
- Extended `probe-section.ts` (W18-W20) — backfill idempotency, section-injection in retrieval, fallback for unsectioned KIs.
- `01-02-SUMMARY.md` with per-AC outcomes, deviations, deferred items, cost-log table.
</objective>

```gherkin

```
