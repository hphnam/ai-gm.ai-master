# Phase 03 — WhatsApp Conversational UX

**Created:** 2026-05-03
**Source:** /paul:discuss 03 (this session)
**Status:** Decisions locked — ready for /paul:plan 03-01

---

## Phase intent

Get the existing chat-v1 `ChatService` ToolLoopAgent (post-Phase-6 default) working *correctly* inside a single WhatsApp thread. The plumbing is already shipped (Infobip inbound webhook + HMAC signature guard + outbound REST + 5MB media download w/ magic-byte validation from v0.2 Plan 03-04; Phone OTP via Twilio Verify from v0.2 Plan 01-03; `ChatService.sendMessage` and `ConversationService` from Phase 6). The missing layer is the **conversational protocol** that sits on top: identity, venue, conversation boundary, onboarding.

This phase **replaces** old Phase 3 (Scheduler + Graph-Aware Notifications) and old Phase 4 (Procedural Runtime), both deferred to v0.4. Reasoning: outbound automation needs a working inbound surface as its substrate. Scheduled notifications that fire into a broken thread protocol just amplify the breakage.

**Why now:** Phase 6 closure exposed D-06-04-A — WhatsApp inbound currently still routes through `ChatV2Service` (multi-agent pipeline) while web `/chat/*` runs on chat-v1. The asymmetry needs to close, and the close requires figuring out conversational shape on WhatsApp anyway. Two birds.

## Locked decisions

### Identity binding

- **D-03-A** — Unknown phone number → bot replies courteously: "I don't recognise this number. Reply with your invite code, or ask your manager for one." No silent reject, no auto-bootstrap.
- **D-03-B** — Bot-initiated linking enabled. User can DM the bot to self-onboard (no requirement to start in `/settings/phone` web flow).
- **D-03-C** — Multi-venue is first-class. Venue switching deserves real UX, not edge-case treatment. Realistic shape: regional managers / multi-site operators are common, not rare.

### Venue context resolution

- **D-03-D** — Sticky-per-conversation venue. Bot picks venue once at conversation start (or carries over from last conversation per D-03-K), sticks until explicit switch.
- **D-03-E** — Switch syntax: `/venue` opens numbered menu (`1) Brixton 2) Camden …`); natural-language detection also works ("switch to brixton", "ask brixton…"). Both supported.
- **D-03-F** — Every bot reply prefixed with `[Venue Name]` so user always sees current context. Visual primary signal of which venue is being scoped.
- **D-03-G** — Out-of-context ad-hoc questions ("what's the ice machine code at brixton?" while sticky=camden) → one-shot override. Bot answers about Brixton this turn, sticky stays on Camden. No "switch?" prompt.

### Conversation boundaries

- **D-03-H** — Conversation boundary triggered ONLY by venue switch. Same-venue threads run continuously without auto-timeout. `/new` exists as manual escape hatch for clean-slate within same venue.
- **D-03-I** — LLM context per turn = current conversation + access to `recall` tool to pull older history on demand. Implementation note: in-context window still gets a fixed sliding truncation (e.g. last 20 turns) for prompt size; `recall` tool fills the gap when the model judges older context relevant. Tool scope: searches past messages on the same `(userId, organizationId)` pair.
- **D-03-J** — Boundary signal: only on explicit `/new` ("👋 Starting fresh — earlier context cleared"). Venue-switch boundaries are silent because the `[Venue]` prefix change is already a visible signal.
- **D-03-K** — Sticky venue persists across conversations. `/new` keeps you on the same venue. Only `/venue` (or natural-language switch) changes it.

### Slash command surface

- **D-03-L** — Six commands total: `/venue` (numbered switch menu), `/new` (clean-slate same venue), `/help` (list commands), `/whoami` (show "you are X, currently asking about [Venue Y]"), `/venues` (list venues you have access to without switching), `/feedback` (capture quick gripe/suggestion → goes to operator inbox).

### Routing

- **D-03-M** — WhatsApp inbound routes through chat-v1 `ChatService.sendMessage` — symmetric with web. `deep_research` tool stays registered on chat-v1 so the agent can escalate per-turn just like on web. **Closes D-06-04-A** (post-Phase-6 asymmetry).

### Onboarding flow

- **D-03-N** — Invite codes generated by manager from `/settings/team` web UI. "Invite via WhatsApp" button → system generates short-lived code (24h TTL) → manager copies and delivers out-of-band (text, in-person, slack, etc.). Code is the entry token.
- **D-03-O** — Verification = WhatsApp OTP. After user submits invite code, bot sends 6-digit code via WhatsApp itself; user replies with it; bot links the phone number to the user account associated with that invite code.
- **D-03-P** — Post-verification welcome = hybrid auto-pick:
  - Single-venue user → bot auto-onboards onto that venue: "Hi [Name], you're on [Venue X]. What can I help with?"
  - Multi-venue user → bot shows numbered list: "Hi [Name]. I have access to N venues for you: 1) Brixton 2) Camden 3) Shoreditch. Which one?"

## Open questions for /paul:plan time

- **Recall tool tuning**: cap on results? scope to `(userId, organizationId)` only or include cross-venue if sticky changed mid-history? probe-tested on what query shape?
- **Sliding-window size**: last 20 turns? 30? Tunable per env? Gut: 20 fits Sonnet's effective working memory and leaves cache headroom; verify in plan probe.
- **Natural-language venue switch detection**: regex keyword match ("switch to X" / "at X" / "for X")? LLM intent classifier? First-pass regex with LLM fallback for ambiguous?
- **Invite code format**: 6-char alphanumeric? UUID? Short-lived JWT signed with `BETTER_AUTH_SECRET`?
- **Rate limiting on the OTP step**: per-phone OTP request limits to prevent SMS-pumping (only relevant if SMS path also used). Pure-WhatsApp OTP is cheap enough that rate limiting is more of a abuse-prevention concern than cost.
- **Conversation row creation timing**: created when user sends first message of a new conversation, or pre-created on venue switch? (Affects conversation list UI on web side if user views past WhatsApp threads.)
- **`/feedback` storage**: new `feedback` table, or write to existing `IncidentLog` with category=feedback?

## Pre-discuss plan-slicing sketch (firm up at /paul:plan)

Likely 2–3 plans:

1. **03-01 — Identity + onboarding flow**
   - `WhatsappInvite` model (manager-generated 24h codes); `/settings/team` "Invite via WhatsApp" UI
   - Webhook handler: unknown-number reply, code-submission flow, OTP send, OTP verify, post-verify hybrid welcome
   - `User.phoneNumber` ↔ `WhatsappInvite.phoneNumber` linkage
   - Probe: full onboarding state machine end-to-end + invite-code expiry + cross-tenant isolation

2. **03-02 — Conversation protocol (venue + boundaries + commands)**
   - WhatsApp-side `WhatsappSession` state (sticky venue, current conversation id, last activity)
   - Slash command parser (6 commands)
   - Natural-language venue switch detection (regex + maybe LLM intent fallback)
   - One-shot venue override mechanic (D-03-G)
   - Conversation row creation on venue switch / `/new`
   - Routing flip: WhatsApp inbound → `ChatService.sendMessage` (closes D-06-04-A)
   - `[Venue]` prefix prepended to outbound replies
   - Probe: command vocabulary, sticky semantics, override behavior, parity-with-web on chat-v1 default path

3. **03-03 — Recall tool + sliding-window context** (may merge into 03-02 if scope feels light)
   - `recall(query?, before?)` tool registered on chat-v1 alongside `find_knowledge` etc.
   - Sliding-window truncation in `ChatService` history-loader (configurable, default 20)
   - Probe: recall correctness on long conversations, window-truncation correctness, recall scoping to `(userId, organizationId)`

## Cost / performance posture

- WhatsApp turn target: $0.01–0.02 (same as web; routes through chat-v1 default).
- OTP message via WhatsApp: free under Infobip trial sender (which we're on); paid tier cost is negligible (~$0.005 per session message).
- Recall tool: only fires when model judges relevant; expected <10% of turns. Postgres FTS over `chat_messages.content` scoped by `(userId, organizationId)` keeps it cheap.
