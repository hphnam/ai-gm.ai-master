<objective>
## Goal

Ship the **identity-binding + onboarding flow** that lets a manager invite a staff member to the WhatsApp chatbot, lets the staff member self-onboard from any phone number by submitting an invite code + verifying via WhatsApp OTP, and routes the post-verification welcome through hybrid auto-pick (single-venue auto, multi-venue numbered menu). Closes the "stranger sends a message" story so 03-02 can layer the conversational protocol (sticky venue, slash commands, routing flip) on top of an authenticated session.

## Purpose

Phase 3 entry point. Without identity binding, every other Phase-3 decision is unreachable — there's no `userId` to thread into `ChatService.sendMessage`, no Membership to scope venue access against, no audit trail. The locked CONTEXT.md decisions D-03-A (unknown-number prompt), D-03-B (bot-initiated linking), D-03-N (manager-issued 24h codes), D-03-O (WhatsApp OTP verification), D-03-P (hybrid welcome) all converge here. SOC-2 CC6.6 cross-tenant discipline carries through end-to-end — invite codes are scoped to the issuing org; OTP attempts are rate-limited; phone numbers are linked atomically.

## Output

- New Prisma models `WhatsappInvite` + `WhatsappOtpAttempt` (additive — no existing column changes; no destructive migration)
- Three new services in `apps/api/src/modules/whatsapp/` — invite generation, OTP send/verify, onboarding state machine
- New REST surface `POST/GET/DELETE /whatsapp/invites` (manager-only, role-guarded)
- Manager UI on `/settings/team` — "Invite via WhatsApp" dialog with code display + copy + revoke
- WhatsApp inbound state machine integrated at `whatsapp.service.handleInbound` boundary — unknown numbers route through onboarding before chat, linked users pass through to existing chat dispatch
- New probe `probe-whatsapp-onboarding.ts` — full state-machine coverage including expiry, revocation, OTP rate limit, cross-tenant isolation
- SUMMARY.md at `.paul/phases/03-whatsapp-conversational-ux/03-01-SUMMARY.md`
</objective>

## AC-12: Timing-safe equality on code + OTP comparison

```gherkin
Given an inbound submission of an invite code OR a 6-digit OTP

Then the comparison uses `crypto.timingSafeEqual(Buffer.from(submitted), Buffer.from(stored))` for codes
```

## AC-14: OTP outbound delivery failure is observable + recoverable

```gherkin
Given user submits a valid invite code
When `whatsapp.adapter.sendOutbound` throws (network, Infobip 5xx, malformed response)
Then the WhatsappOtpAttempt row is flipped to `status="failed_send"` (NOT left in "pending" — that would block re-issuance)
And the user receives a fallback inbound reply: "Couldn't send your code. Try replying with your invite code again in a moment." — sent on a best-effort basis (if outbound is fully down, even this reply may fail; that is acceptable)
And `whatsapp_otp.send_failed` is logged with {inviteId, errorCode?, latencyMs} — never the OTP plaintext
And subsequent code submissions for the same invite within rate-limit window can issue a fresh OTP attempt (the failed_send row does NOT count as an "active" attempt for the re-issuance guard)
And rate-limit accounting still increments — failed sends DO count against the per-phone WHATSAPP_OTP_RATE_LIMIT_PER_HOUR=3 ceiling (otherwise an attacker could induce send failures to bypass rate-limit)
```
