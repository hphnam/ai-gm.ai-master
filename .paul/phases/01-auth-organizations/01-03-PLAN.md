<objective>
## Goal
Ship phone-number linking end-to-end â€” an authenticated user enters an E.164 phone number in `/settings/phone`, receives a 6-digit SMS OTP via Twilio Verify, submits the code, and on success has `User.phoneNumber` + `User.phoneVerifiedAt` persisted. Any subsequent sign-in by the same user shows the phone as verified. Concurrent sessions on different devices share the same verified state. A user can unlink their number to link a different one. Numbers are unique across all users â€” a number already linked to another account fails explicitly. Rate limits prevent SMS abuse.

## Purpose
Plan 01-01 shipped auth + orgs and Plan 01-02 shipped team invitations. Phase 3 (WhatsApp Integration) needs a way to map an inbound `From: +447...` WhatsApp number back to a `User â†’ Organization â†’ Venue` chain â€” phone-linking is the only primitive that enables that lookup without forcing every WhatsApp message through a magic-link dance. This plan turns the dormant `User.phoneNumber` / `User.phoneVerifiedAt` columns (already in schema from 01-01) into a working verification flow, and is the last v0.2 Phase 1 plan before Phase 2 (Document Ingest UI).

Per-message SMS costs make rate-limiting a higher priority than the email equivalent (01-02). Twilio Verify is chosen over rolling our own OTP storage because (a) Twilio handles code generation / expiry / attempt counts server-side, removing a whole class of cryptographic bugs from our surface, (b) their per-number and per-IP rate limits are production-grade out of the box, (c) swap-out to alternate verification channels (voice, WhatsApp OTP) is a Twilio config change not a code rewrite.

## Output
- NestJS `PhoneModule` exposing:
  - `POST /auth/phone/send` â€” authenticated; body `{ phoneNumber }` (E.164). Initiates Twilio Verify start. Returns `{ ok: true, expiresInSeconds: 600 }` on success. Calls Twilio Verify check; on approval sets `User.phoneNumber` + `User.phoneVerifiedAt` in a single transaction with uniqueness check. Returns `{ ok: true, phoneNumber, phoneVerifiedAt }`. <!-- audit-added M1: verify ALSO requires a matching pending-verification entry in PhoneService.pendingVerifications for the session userId â€” blocks cross-session code-claim attacks -->

## AC-1: Authenticated user sends verification code

```gherkin
Given an authenticated session for a user with no linked phone
When the client POSTs /auth/phone/send with { phoneNumber: "+447700900123" }
Then Twilio Verify is called with channel='sms' and the number
And the response body is { ok: true, expiresInSeconds: 600 }
And a structured log `phone.verify_sent` is emitted with { userId, phoneHash, requestId }
And the raw phoneNumber is NEVER written to any log (only sha256-prefix-16 hash)
```

## AC-13: Cross-session code-claim attack blocked (audit-added M1)

```gherkin
Given user A is authenticated in session Sa and POSTs /auth/phone/send with { phoneNumber: "+447700900123" }
And the Twilio Verify service accepts the start (console-mode code deterministically computable)
And user B is authenticated in a DIFFERENT session Sb (different userId)
When user B POSTs /auth/phone/verify with { phoneNumber: "+447700900123", code: <correct-code> }
Then the response is 400 with { error: "phone-verification-failed" }
And user B's User row phoneNumber is unchanged (null)
And a structured log `phone.cross_session_blocked` is emitted { attemptingUserId, phoneHash }

Rationale: without this guard, any authenticated user who sees/steals/guesses a 6-digit
verification code could claim a number another user initiated. PhoneService owns a
pendingVerifications map keyed by sessionUserId; the verify endpoint requires the
entry exists, the phoneNumber matches, and the entry is not expired (10-min TTL).
```

## AC-14: Phone change requires explicit unlink (audit-added M2)

```gherkin
Given an authenticated session for a user with User.phoneNumber = "+447700900111" already set
When the client POSTs /auth/phone/send with { phoneNumber: "+447700900222" } (a different number)
Then the response is 409 with { error: "phone-change-requires-unlink" }
And Twilio Verify is NOT called
And a structured log `phone.change_without_unlink_blocked` is emitted { userId, currentPhoneHash, attemptedPhoneHash }

Rationale: phishing-style number swap ("verify this new number") could silently
redirect future WhatsApp routing to an attacker-controlled number. Forcing explicit
unlink makes the state change visible to the user in two steps.
```

## AC-15: TWILIO_DRIVER_OVERRIDE='disabled' kill-switch (audit-added M3)

```gherkin
Given TWILIO_DRIVER_OVERRIDE=disabled is set in the environment
When any authenticated session POSTs /auth/phone/send OR /auth/phone/verify
Then the response is 503 with { error: "phone-service-unavailable", details: { reason: "disabled" } }
And Twilio Verify REST API is NOT called (no outbound HTTP)
And NO pendingVerifications entry is created
And a structured log `phone.driver_disabled` is emitted { userId, path }

Rationale: operator kill-switch for abuse events or Twilio credential compromise
without redeploying code. Flag in .env, bounce the API, SMS spend halts immediately.
```

## AC-16: Idempotent unlink (audit-added M9)

```gherkin
Given an authenticated session for a user with User.phoneNumber = null (no phone linked)
When the client DELETEs /auth/phone
Then the response is 200 with { ok: true }
And NO structured log `phone.unlinked` is emitted (no state change)
And the DB query is NOT issued (short-circuit check)

Rationale: DELETE is idempotent per REST convention. UI can always call DELETE
without pre-checking status â€” simpler client code, no phone-not-linked error code.
```
