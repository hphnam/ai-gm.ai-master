<objective>
## Goal
Ship organisation invitations end-to-end — owner/manager creates an invite for an email address + role, the invitee receives an email with a signed accept-link, clicks it, signs up (or signs in) with the invited email, and joins the organisation with the assigned role. No new org is created on invite-accept. Staff role cannot create invitations. Expired or already-accepted invitations fail with explicit error codes.

## Purpose
Plan 01-01 shipped auth + orgs but every sign-up creates a brand-new workspace. Real venue managers need to bring their team into their existing org — that's the whole multi-tenant premise of v0.2. This plan turns the dormant `Invitation` table (already in schema from 01-01) into a working flow, and is the prerequisite for both Plan 01-03 (phone linking — invitee links their number after join) and Phase 3 WhatsApp onboarding (staff invited by manager → phone-verified → WhatsApp-ready).

## Output
- NestJS `InvitationsModule` exposing:
  - `POST /org/invitations` — owner/manager creates invite `{ email, role }` → returns `{ invitation, inviteUrl }` (inviteUrl for dev visibility)
  - `GET /org/invitations` — owner/manager lists all pending + recent invitations for their org
  - `DELETE /org/invitations/:id` — owner/manager cancels pending invite (sets status='revoked')
  - `GET /org/invitations/:id/preview` — unauthenticated preview (invite recipient lookup) returning `{ email, organizationName, role, status, expiresAt }` or 404/410
  - `POST /org/invitations/:id/accept` — **authenticated** accept; invitee's session email must match invitation email; on success creates OrganizationMember, sets status='accepted', returns updated session active org
- `MailService` with Resend driver (when `RESEND_API_KEY` is present) + console fallback for dev
- Extended `assertAuthEnv` reading optional `RESEND_API_KEY` + `MAIL_FROM` (fail-soft → console fallback)
- Next.js `/settings/organization` page (owner/manager only) — invite form (email + role dropdown) + invitation list (status badges, copy-link, revoke button)
- Next.js `/auth/accept-invitation/[id]` page handling three states: signed-out (redirect to sign-in w/ `?next=`), signed-in wrong-email (explicit error), signed-in matching-email (one-click accept → redirect to `/chat` with new active org)
- `probe-auth.ts` extended P12–P17 covering: create, staff-forbidden, accept happy path, expired, already-accepted, cross-org list isolation
- `.env.example` + `packages/types/src/api.ts` append `invitation-not-found` / `invitation-expired` / `invitation-already-accepted` / `invitation-email-mismatch` / `mail-send-failed` error codes
</objective>

## AC-4: Expired and already-accepted invitations fail cleanly  <!-- audit-added M6: unified expired response -->

```gherkin
Given an invitation where expiresAt < now() AND status = "pending"
When the client POSTs /org/invitations/{id}/accept
Then the API returns 410 { error: "invitation-expired" }
And the invitation status is updated to "expired"

# audit-added M6 — expired status is ALWAYS 410, never 404 on retry (prev draft flipped to 404 after first expiry)
Given an invitation where status = "expired" (from prior accept attempt flipping pending→expired)
When the client POSTs /org/invitations/{id}/accept again
Then the API returns 410 { error: "invitation-expired" } (same as first expiry — not 404)

Given an invitation where status = "accepted"
When the client POSTs /org/invitations/{id}/accept again
Then the API returns 409 { error: "invitation-already-accepted" }

Given an invitation where status = "revoked"
When the client POSTs /org/invitations/{id}/accept
Then the API returns 404 { error: "invitation-not-found" } (enumeration-safe: revoked is indistinguishable from never-existed to the invitee — prevents leaking that a targeted email was invited-then-cancelled)

Given an id that does not exist on any org
When the client hits any /org/invitations/:id endpoint (accept, preview, revoke)
Then the API returns 404 { error: "invitation-not-found" }

# audit-added M6 — response matrix (pin it down to prevent drift)
# | status      | accept returns         | preview returns        | revoke returns         |
# | pending     | 200 + accept           | 200 + preview          | 200 + ok               |
# | accepted    | 409 already-accepted   | 200 + preview (shows "Already accepted") | 409 already-accepted |
# | revoked     | 404 not-found          | 404 not-found          | 404 not-found          |
# | expired     | 410 expired            | 410 expired            | 404 not-found          |
# | not-found   | 404 not-found          | 404 not-found          | 404 not-found          |
```

## AC-11: Per-org pending-invitation cap prevents abuse  <!-- audit-added M7 -->

```gherkin
Given an org already has MAX_PENDING_INVITATIONS_PER_ORG = 50 invitations with status='pending' AND expiresAt > now()
When an owner/manager POSTs /org/invitations with any valid body
Then the API returns 429 { error: "invitation-limit-reached", details: { pending: 50, limit: 50 } }
And no new Invitation row is created
And a structured log `invitation.limit_reached { organizationId, pending, limit }` is emitted at warn level

# Reason: without this cap, a compromised owner account (or malicious insider) could invite 10,000 addresses in minutes, burning Resend quota AND spamming targets AND giving attackers 10,000 accept-URLs to abuse. 50 is generous for legitimate SME teams, trivially raisable via code constant.
# Lazy-GC semantics: expireStaleInvitations() runs at the start of createInvitation to flip stale pending→expired, so the cap counts only TRULY pending invites.
# Constant lives in @gm-ai/types (mirrors v0.1 MAX_RETAG_ATTEMPTS pattern) — one-place-tunable, type-visible to consumers.
```
