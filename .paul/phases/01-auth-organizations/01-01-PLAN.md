<objective>
## Goal
Ship the auth foundation that makes every existing endpoint in the gm-ai monorepo require an authenticated session tied to an Organization: better-auth + `organization` plugin wired against the Prisma schema, NestJS AuthGuard + role guards applied to every existing controller, Next.js `/auth/sign-up` + `/auth/sign-in` flows (email + password, shadcn defaults), and a proper Prisma migration that backfills seeded Crown + Anchor venues under a new "Demo Organization" — without breaking the v0.1 chat + debug loop for an authenticated user.

## Purpose
v0.1 POC is an unauthenticated demo. v0.2 needs real users (managers + staff) before any WhatsApp integration can target them. This plan puts the multi-tenant foundation in place first — every subsequent v0.2 plan (invite flow in 01-02, phone linking in 01-03, doc upload in Phase 2, WhatsApp in Phase 3) depends on having `User`, `Organization`, `OrganizationMember`, and `@CurrentOrg()` available. This is the trust boundary that WhatsApp's phone-number lookup, doc upload's tenant isolation, and Coolify's auth-gated deployment all rely on.

## Output
- Prisma schema + migration adding: User, Session, Account, VerificationToken (better-auth managed), Organization, OrganizationMember, Invitation, `User.phoneNumber String? @unique` (nullable; verification flow deferred to Plan 01-03), `Venue.organizationId` FK
- Idempotent data-migration script attaching existing Crown + Anchor + their KnowledgeItem + mock_* rows to a new "Demo Organization" owned by `ryan@ryanhelmn.dev`
- NestJS `AuthModule` exporting: `AuthGuard`, `RoleGuard`, `@CurrentUser()`, `@CurrentOrg()`, `@RequireRole('owner'|'manager')` decorators
- Every existing controller (`ChatController`, `SuggestionsController`, `FeedbackController`, `VenuesController`, `DebugController`) behind `AuthGuard` + tenant-scoped at the service layer via `@CurrentOrg()` read + repository-layer `withOrgScope()` helper
- Next.js `/auth/sign-up`, `/auth/sign-in` routes with shadcn default forms; root-layout auth gate redirects unauth users to `/auth/sign-in?redirect=<url>`
- Extended `probe-api.ts` (A23–A26) guarding unauth + cross-org + role-gate enforcement; new `probe-auth.ts` running a full sign-up → cross-org-isolation → sign-out flow end-to-end
- `.env.example` gains `BETTER_AUTH_SECRET`, `BETTER_AUTH_URL`, `DEMO_USER_PASSWORD`, `DEMO_USER_EMAIL`
- Human UAT checkpoint: Ryan opens localhost:3000 → redirected to /auth/sign-up → signs up → lands on /chat (his new org) → existing chat loop still works; second browser profile signs up → different org → sees no Crown/Anchor
</objective>
