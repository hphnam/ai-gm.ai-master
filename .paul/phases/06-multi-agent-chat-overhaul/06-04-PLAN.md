<objective>
## Goal
**Delete chat-v1 entirely.** Move feature parity (image upload, streaming, conversation list/fetch/delete) onto chat-v2; migrate the WhatsApp consumer from `ChatService` to `ChatV2Service`; drop the `Organization.chatV2Enabled` flag column; flip the controller to route everything through chat-v2; clear a quality gate (probe-eval ≥80% on 12-query harness + manual UAT ≥18/20 amazing on canary venue); then `rm -rf apps/api/src/modules/chat/`. Rollback path is `git revert` only — user explicitly accepted no-flag-rollback risk.

## Purpose
06-03 shipped pipeline breadth behind the per-org flag (default false). The flag itself plus the chat-v1 module is dual-maintenance burden: image uploads, streaming, conversation list/fetch/delete, and WhatsApp inbound all still flow through chat-v1, so chat-v2 has never had a feature-complete production load. CONTEXT.md D-06-G ("Feature-flag cutover with empirical quality gate + 2-week soak") was superseded 2026-05-01 by user decision "bin the flag, fully migrate" — direct deletion replaces flag flip; quality gate becomes a pre-deletion checkpoint. This plan executes that decision.

## Output
- chat-v2 absorbs all chat-v1 surface area (1 multimodal endpoint, 1 streaming endpoint, 3 conversation endpoints, WhatsApp consumer)
- `Organization.chatV2Enabled` column dropped via Prisma migration
- `apps/api/src/modules/chat/` directory deleted; `app.module.ts` no longer imports `ChatModule`
- Probe-eval extended 6-query → 12-query harness spanning all 3 modes (lookup/reasoning/incident); pass-rate ≥80% gate
- Manual UAT checklist (20 questions on canary venue) ≥18 amazing + zero shit
- `apps/api/src/modules/chat-v2/06-04-SUMMARY.md` (via /paul:unify) documenting actuals
</objective>

## AC-1: Image-upload endpoint on chat-v2 with magic-byte hardening
```gherkin
Given POST /chat/messages/with-image arrives with multipart `file` (image) + `userMessage` + `venueId` + optional `conversationId`
When chat-v2.controller.ts receives the request
Then file passes ALL FOUR layers: (1) MIME allowlist (image/jpeg|png|webp|gif), (2) declared-MIME→signature magic-byte match via re-used `apps/api/src/modules/ingest/image-magic-bytes.ts` helper, (3) MAX_IMAGE_BYTES_BY_MIME ceiling, (4) multer per-request 15MB cap
And on validation failure → 400 with `ApiErrorResponse { error: 'unsupported-mime'|'corrupt-bytes'|'payload-too-large' }` matching Phase 4 04-01 contract
And on success → ChatV2Service.sendMessage receives `attachment: { mediaType, base64 }` and the multimodal turn ships through the same Triage → Researchers → Analyser → Writer → Critic pipeline as text turns

```
