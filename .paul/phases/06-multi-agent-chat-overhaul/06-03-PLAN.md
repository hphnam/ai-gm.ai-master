<objective>
## Goal
Ship pipeline **breadth** for chat-v2: four new researcher specialists (Ops, People, Tabular, Venue) with shaped tools, parallel fan-out in the orchestrator, and Triage prompt rewritten to dispatch per-mode researcher subsets вЂ” all behind the existing `chatV2Enabled` org flag with zero regression on 06-01/06-02 surface.

## Purpose
06-02 shipped pipeline depth (Analyser + Critic + reasoning/incident modes) using a single Docs researcher. The pipeline is structurally ready for multiple researchers (TriageOutput already lists all 5 names; orchestrator dispatch is researcher-list-driven), but `chat-v2.service.ts` hard-codes `[docsResult.finding]`. CONTEXT.md D-06-A names five researchers as the architectural contract; D-06-B says Venue runs always on reasoning + incident. Without breadth, the system can never deliver "proactive" reasoning вЂ” Venue briefing is the structural source of that property. This plan wires the breadth.

## Output
- 4 new NestJS researcher providers conforming to the existing `research(brief, ctx) в†’ { finding, usage, voyageCalls }` shape established by `DocsResearcher`
- 2 new shaped tools (`get_person`, `get_venue_briefing`) + thin Ops/Tabular tool wrappers re-using existing services
- Triage prompt + stub rewritten to dispatch per-mode researcher subsets with Venue always-on for reasoning + incident
- Orchestrator parallel fan-out via `Promise.all` over `triage.researchersToDispatch`, gracefully degrading single-researcher failures to partial findings
- Probe-chat-v2 extended V51-V80+ covering parallel fan-out, per-researcher cross-tenant isolation, mode-dispatch routing, and stub-mode determinism
- `apps/api/src/modules/chat-v2/06-03-SUMMARY.md` (via /paul:unify) documenting actuals
</objective>
